## 1. ConfigService — 配置服务

### 1.1 需求

- ✅ **配置扫描与索引** — 自动扫描 `group/`、`skills/`、`tools/` 下的 Markdown 文件，提取 YAML frontmatter 并建立多维度索引（role_id / process_id / skill_id / tool_id）
- ✅ **角色 Soul 文件查询** — 支持别名解析（如 `suri-dev` → `suri_dev`），自动回退到兼容路径
- ✅ **核心角色自动重建** — 若 `suri`、`suri_dev`、`suri_hr`、`suri_review`、`suri_stats` 的 Soul 文件缺失，自动生成默认模板并写入磁盘
- ✅ **部门与角色管理 API** — `list_departments()`、`get_department_lead()`、`get_role_keywords()`、`get_role_capabilities()`、`list_role_skills()`、`get_skill_detail()`
- ✅ **动态角色路由构建** — `sync_group_function()` 可自动生成 `group/group_function.md`（角色能力速查表）
- ✅ **别名解析体系** — `_ROLE_ALIASES` 映射 + `resolve_role_id()` canonical id 保证
- ✅ **工具注册表读取** — 从 `tool_registry.json` 读取工具元数据

### 1.2 待完善

- 🔄 YAML 配置加载（`get_model_pool()`、`get_memory_config()`、`get_telegram_config()`）仅做文件读取，未做结构校验或热重载逻辑

## 2. MemoryService — 记忆服务

### 2.1 需求

- ✅ **角色级独立 SQLite 存储** — 每个角色拥有独立的 `role.db`，启用 WAL 模式支持并发读写
- ✅ **多表结构** — sessions、tasks、messages、approvals、changelogs、statistics、experiences（经验日志表）
- ✅ **会话与任务管理** — `create_session()`、`close_session()`、`create_task()`、`update_task_status()`、`increment_retry()`，支持按 session_id 过滤消息实现多用户隔离
- ✅ **消息管理** — `save_message()`、`get_session_messages()`（JOIN tasks 表按 session 隔离）、`get_task_messages()`
- ✅ **经验系统** — `save_experience()`、`get_experiences()`、`get_experience_stats()`
- ✅ **角色私人记忆文件** — `save_role_memory()`、`list_role_memories()`（按修改时间倒序）
- ✅ **Insight 经验文件** — `save_role_insight()`（带 YAML frontmatter）、`list_role_insights()`、`_build_insight_markdown()`、`_parse_insight_frontmatter()`
- ✅ **上下文注入** — `get_recent_insights_for_context()` 支持按时间、confidence、关键词粗排，并自动截断字符数
- ✅ **触发计数更新** — `update_insight_trigger()` 解析 frontmatter 并重建文件，避免正则脆弱性
- ✅ **统计表** — `save_statistic()`、`get_statistics()`（Token 消耗、文件创建、任务完成）
- ✅ **聚合查询** — `get_all_tasks()`（遍历所有角色 role.db 聚合 tasks 表）、`get_pending_approvals()`（查询 approvals 表）

### 2.2 待完善

- 🔄 **记忆归档/遗忘策略** — 虽然表结构支持，但没有自动清理或压缩旧数据的定时任务

## 3. LoggerService — 日志服务

### 3.1 需求

- ✅ **分类日志写入** — 按 `logs/categories.yaml` 定义的分类（runtime / error / schedule / role / system / statistics / tool_calls）写入独立目录
- ✅ **按天轮转** — 文件名格式 `suri-{YYYY-MM-DD}.log`，自动追加
- ✅ **结构化 JSON 日志** — `_write_json_log()` 写入 `.jsonl`（用于统计和工具调用审计）
- ✅ **业务事件快捷方法** — `log_startup()`、`log_shutdown()`、`log_user_input()`、`log_task_created()`、`log_task_dispatched()`、`log_model_call()`、`log_token_usage()`、`log_file_created()`、`log_task_completed()`、`log_tool_call()`、`log_learning()`、`log_learning_error()`
- ✅ **日志查询** — `get_today_logs()` 返回当日各分类日志文件路径
- ✅ **Token 消耗统计** — `log_token_usage()` 记录 prompt_tokens / completion_tokens / total_tokens

### 3.2 待完善

- 🔄 **日志级别过滤** — 当前所有级别均写入，没有运行时动态过滤或丢弃低级别日志的机制

## 4. SecurityService — 安全服务

### 4.1 需求

- ✅ **权限检查委托** — 调用 `FileOwnershipRule` 和 `SecurityRule` 进行文件所有权与安全审批校验
- ✅ **核心角色 Soul 文件保护** — 禁止非 `suri_hr` 修改核心角色的 Soul 文件
- ✅ **豁免路径优先** — `_is_exempt()` 将 `resources/logs/`、`resources/cache/`、`resources/temp/` 等临时目录标记为自动操作无需审批
- ✅ **审批令牌验证** — `validate_approval_token()` 委托 `SecurityRule.validate_token()`
- ✅ **变更报告与签发** — `create_change_report()`、`issue_approval_token()`、`register_approval()`、`revoke_approval()`
- ✅ **综合前置检查** — `pre_file_change_check()` 串联豁免→权限→监控范围→审批四层逻辑

### 4.2 待完善

- 🔄 **离线代理机制** — `check_offline_proxy()` 仅返回字符串标记，未实际执行代理审批逻辑

## 6. ModelManager — 模型管理器

### 6.1 需求

- ✅ **配置持久化** — `model_config.json` 读写，支持添加、删除、设置默认模型
- ✅ **首次运行引导** — `setup_wizard()` 两级菜单（品牌 → 型号），支持自定义模型
- ✅ **API Key 自动测试** — `_test_api_key()` 依次测试 primary 和 fallbacks，处理 401/429/403/503/500 等状态码
- ✅ **多格式模型调用** — OpenAI 兼容格式与 Anthropic 格式的异步调用
- ✅ **Token 用量返回** — `chat_with_usage()` 返回 prompt_tokens / completion_tokens / total_tokens
- ✅ **流式输出** — `chat_stream()` 支持 SSE 逐 token 返回（OpenAI 和 Anthropic 均实现）
- ✅ **自动降级** — `chat()` / `chat_with_usage()` 按优先级遍历候选模型，失败自动切换
- ✅ **智能任务路由** — `select_model_for_task()` 从任务内容推断所需能力（coding / vision / reasoning / long_context / fast），按完全匹配→部分匹配→默认模型三级选择，并优先低成本模型
- ✅ **httpx 异步客户端** — 连接池复用，tenacity 指数退避重试（3 次）
- ✅ **环境变量同步** — `_persist_config()` 同时更新 `.env`
- ✅ **模型分类体系** — 6 大类型（chat / coder / vision / reasoning / long_context / fast），`ModelConfig` 新增 `model_type` 字段
- ✅ **成本分级** — `cost_tier` 字段（free / low / medium / high / premium）

### 6.2 待完善

- 🔄 `presets.json` 异常处理 — 若 presets.json 缺失，使用空字典回退，模型能力标签可能不准确
