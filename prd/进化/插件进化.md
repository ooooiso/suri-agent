# 插件进化

> 定义 Plugin 维度的进化机制 —— 插件如何自我分析、学习升级、自修改代码。

> **核心原则**：每个插件也是 Agent，拥有学习能力，但所有代码自修改须用户确认。

---

## 一、插件进化的维度

插件可以从以下维度进化：

| 维度 | 说明 | 示例 |
|------|------|------|
| **自身能力** | 插件学习如何更好地完成自己的职责 | 日志插件学到新的日志聚合模式 |
| **暴露的工具** | 插件新增/更新/废弃暴露的 MCP 工具 | code_tool 新增批量搜索工具 |
| **事件处理** | 插件订阅新事件或优化事件处理逻辑 | interrupt_handler 学习到新的中断模式 |
| **性能优化** | 插件分析自身性能瓶颈并优化 | config_service 学习到缓存策略 |
| **接口变化** | 插件更新 manifest.json 暴露声明 | plugin declares new api endpoint |

## 二、插件进化流程

```
插件自分析（role_learner 系统级/插件自身数据）
    │
    ├── 插件收集自身运行数据
    │   ├── 响应时间分布
    │   ├── 错误率统计
    │   ├── 事件处理模式
    │   └── 工具调用热度
    │
    ├── 识别改进机会
    │   ├── 重复模式 → 可工具化、自动化
    │   ├── 性能瓶颈 → 可优化
    │   ├── 能力缺口 → 可新增
    │   └── 事件重复订阅 → 可合并
    │
    ▼
插件生成方案
    ├── 方案一：改进已有能力（hot reload 级）
    ├── 方案二：新增工具暴露（warm reload 级）
    └── 方案三：重构插件逻辑（cold reload 级）
    │
    ▼
suri 评估方案
    ├── 确认方案不违反安全规范
    ├── 确认方案与系统架构一致
    └── 确认方案有回滚策略
    │
    ▼
用户确认
    │
    ▼
执行自修改
    ├── 创建备份（~/.suri/backup/）
    ├── 应用变更
    ├── 执行验证
    ├── 更新 manifest.json 版本
    └── 广播 plugin.updated
    │
    ▼
热更新生效
    ├── hot：代码立即热替换
    ├── warm：等待当前事件处理完成
    └── cold：标记重启时生效
```

## 三、插件进化的触发方式

| 触发方式 | 频率 | 说明 |
|---------|------|------|
| **定期自分析** | 每天/每 1000 次事件 | role_learner 分析插件运行数据 |
| **用户触发** | 用户要求 | 用户要求插件增加新能力 |
| **事件驱动** | 检测到模式 | 插件检测到可优化的模式 |
| **suri 触发** | suri 要求 | suri 发现插件能力缺口 |
| **系统提示** | update_manager 检测 | upgrade_manager 检测到有更好的实现 |

---

## 四、插件进化 vs 开发者手动修改

| 方面 | 插件自进化 | 开发者手动修改 |
|------|-----------|---------------|
| 速度 | 快，无需打开编辑器 | 慢，需要编辑代码 |
| 模式 | 以已有模式为基础优化 | 可以全新实现 |
| 创造性 | 有限，基于 pattern | 无限制 |
| 安全性 | 经过 suri 和用户双重确认 | 开发者自行确认 |
| 适用 | 增补工具/优化性能/调整逻辑 | 重大架构变更/新插件开发 |

## 五、插件升级管理

### 版本和回滚

```
每次自修改：
  1. 当前代码备份至 ~/.suri/backup/{plugin_name}/{timestamp}/
  2. 更新 manifest.json version 字段
  3. 备份旧 manifest.json

回滚操作：
  1. suri 选择回滚目标版本
  2. 停止事件处理
  3. 还原代码 + manifest.json
  4. 更新版本标记（回滚到 v{major}.{minor}）
  5. 恢复事件处理
  6. 广播 plugin.rollback
```

### 插件进化与四维协同

插件进化后：
1. **暴露新工具** → 触发 `tool.registered` → 所有角色收到
2. **事件订阅变化** → 触发 `event.contract_updated` → EventBus 调整分发
3. **能力更新** → 触发 `plugin.updated` → 能力索引增量重建
4. **Soul 不适应** → 通知 suri 更新相关角色的 Soul

## 六、限制与约束

1. **插件不可修改系统核心插件（suri_core）**
2. **插件不可修改其他插件代码**（只能触发协作事件）
3. **所有暴露的工具变更必须更新 manifest.json**
4. **中断性变更只能通过 cold reload**
5. **同一插件的并发升级被禁止**
6. **升级方案必须包含验证步骤和回滚策略**
7. **非注册工具不可公开暴露**
