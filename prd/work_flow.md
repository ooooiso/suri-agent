# 项目的工作流

> 描述项目级别的完整工作流程。核心变化：**项目总监**作为项目附带的调度角色，用户直接在项目群组与项目总监交互，suri 只在项目组创建和缺少角色时介入。

---

## 核心概念

### 项目组（Project Group）

每个复杂项目 = 3 个要素：

```
works/{project_id}/          # 项目工作目录
├── .meta.json              # 项目元数据
├── prd.md                  # 项目需求文档
├── plan/                   # 任务规划
├── output/                 # 成果输出
└── logs/                   # 项目日志

+ Telegram 项目群组         # 外部沟通空间
  └─ chat_id 记录在 .meta.json

+ 项目总监角色              # 项目附带的调度角色
  └─ role_id: {project_id}_director
```

### 项目总监（Project Director）

- **性质**：项目附带角色，每个复杂项目自动创建一个
- **职责**：项目群内任务调度、进度跟踪、结果汇总、角色协调
- **权限**：项目内最高调度权限，可调用 task_planner / task_scheduler / agent_registry
- **Soul 生成**：由 suri 根据项目需求自动生成，用户确认
- **通信**：在项目 Telegram 群组中以 Bot 身份活跃

### 角色分工

| 角色 | 场景 | 职责 |
|------|------|------|
| **suri** | 项目组创建、缺少角色时 | 分析需求、建议建项目组、生成项目总监 Soul、请求用户创建新角色 |
| **项目总监** | 项目运行期间 | 接收群内任务、调度执行、跟踪进度、汇总结果 |
| **实现角色** | 执行具体任务 | 接收项目总监分配的子任务、执行、汇报 |

## 1. 项目组创建流程

```
用户：我要做一个电商网站
    │
    ▼
suri 接收并分析（通过 access，私聊或默认群）
    │
    ▼
判断为复杂项目
    │
    ▼
suri 向用户确认：
"这个项目比较复杂，建议成立项目组：
 1. 在 works/ 建立项目目录
 2. 在 Telegram 创建项目群组【电商网站】
 3. 创建项目总监角色负责调度
 是否确认？"
    │
    ▼
用户确认
    │
    ▼
suri 执行创建：
  ├─ filesystem: 创建 works/ecommerce-site/ + .meta.json
  ├─ access/telegram: 创建 Telegram 群组，邀请用户和 bot
  ├─ role_manager: 创建项目总监角色（先生成 Soul 草案，用户确认）
  └─ agent_registry: 初始化项目主 Agent
    │
    ▼
项目总监角色就绪
    │
    ▼
suri 在群组中介绍项目总监：
"【电商网站】项目组已成立。
项目总监 @ecommerce_director 负责项目调度。
请在群组中直接向项目总监说明任务。"
    │
    ▼
suri 退出项目日常调度（退居二线）
```

**项目总监 Soul 生成流程**：

```
suri 根据项目需求生成 Soul 草案
    │
    ▼
向用户呈现：
"项目总监角色草案：
- 昵称：电商总监
- 职责：负责电商网站项目的整体调度..."
    │
    ▼
用户确认 / 修改
    │
    ▼
role_manager.create_role() 创建角色
```

## 2. 项目群内任务执行流程

```
用户在 Telegram 项目群组中：
"@ecommerce_director 帮我设计一个用户登录页面"
    │
    ▼
access/telegram 捕获 @提及
    │
    ▼
路由规则：
  ├─ @项目总监 ──▶ 直接路由给项目总监角色
  ├─ @实现角色 ──▶ 直接路由给对应角色
  └─ 无 @ ──▶ 默认路由给项目总监
    │
    ▼
项目总监接收任务
    │
    ▼
项目总监分析任务：
  ├─ 简单任务 ──▶ 直接分配给自己或调用工具执行
  ├─ 需分解 ──▶ 调用 task_planner.plan()
  └─ 需其他角色 ──▶ 查看项目成员，分配给对应角色
    │
    ▼
项目总监在群中回复：
"收到，正在安排 suri_design 设计登录页面，预计 30 分钟。"
    │
    ▼
创建子 Agent（agent_registry.create_sub_agent）
    │
    ▼
task_scheduler 调度执行
    │
    ▼
实现角色执行步骤
    │
    ▼
每步完成，项目总监在群中汇报进度
    │
    ▼
全部完成，项目总监在群中交付结果
```

## 3. 缺少角色处理流程

```
项目总监分析任务后，发现缺少实现角色
    │
    ▼
项目总监汇报 suri（role_comm.send 给 suri）
    │
    ▼
suri 接收："电商网站项目缺少前端开发角色"
    │
    ▼
suri 请求用户决定（通过 access，私聊）：
"项目需要前端开发角色，请提供：
1. 昵称（如：前端开发）
2. 角色定义（一句话描述职责）

或直接输入：/create_role 前端开发 "负责网页前端开发，使用 React/Vue"
"
    │
    ▼
用户输入昵称和定义
    │
    ▼
suri 丰富 Soul 内容，生成完整角色草案
    │
    ▼
向用户呈现草案，确认
    │
    ▼
role_manager.create_role() 创建角色
    │
    ▼
项目总监将新角色引入项目组
    │
    ▼
继续任务调度
```

**角色创建简化流程**：

```
终端对话 / Telegram 私聊
    │
    ▼
用户：/create_role 昵称 "角色定义"
    │
    ▼
suri 接收，调用 llm_gateway 丰富 Soul：
  ├─ 根据昵称和定义生成完整职责描述
  ├─ 生成能力清单
  ├─ 生成关键词
  └─ 生成工作方法论
    │
    ▼
suri 向用户呈现完整 Soul 草案
    │
    ▼
用户确认 / 修改
    │
    ▼
role_manager 创建角色目录 + Soul 文件
    │
    ▼
角色就绪，可立即被项目总监调用
```

## 4. 多角色协作流程（项目群内）

```
项目总监分解复杂任务
    │
    ▼
调用 task_planner 生成协作规划
    │
    ▼
为每个子任务创建子 Agent
    │
    ▼
各实现角色在项目中执行
    │
    ├─ 并行步骤：同时执行，通过 role_comm 交换中间结果
    │   └─ 结果同步到 Telegram 群（项目总监汇总后播报）
    │
    └─ 串行步骤：按依赖顺序执行
    │
    ▼
每 N 分钟或关键节点，项目总监在群中播报进度
    │
    ▼
如有角色受阻：
  ├─ 项目总监先尝试协调解决
  ├─ 无法解决 ──▶ 汇报 suri
  └─ 或升级给用户决策（通过 access 在群中呈现选项）
    │
    ▼
所有子任务完成
    │
    ▼
项目总监汇总成果，在群中交付
```

## 5. 项目进度跟踪

```
用户随时在项目群中查询：
"@ecommerce_director 进度如何？"
    │
    ▼
项目总监查询 agent_registry
    │
    ▼
汇总各 Agent 状态：
  ├─ 设计阶段：已完成 ✅
  ├─ 前端开发：进行中 ▶ 3/5 步骤
  └─ 后端接口：等待中 ⏸ 依赖前端设计
    │
    ▼
在群中回复进度报告
```

## 6. 项目异常处理

```
角色执行受阻
    │
    ▼
项目总监先尝试协调解决
    │
    ├── 能解决 ──▶ 重新调度或调整规划
    │
    └── 无法解决 / 缺少关键能力
        │
        ▼
    汇报 suri
        │
        ▼
    suri 判断：
      ├─ 需新角色 ──▶ 进入【3. 缺少角色处理流程】
      ├─ 需用户决策 ──▶ 通过 access 在群中呈现选项
      └─ 技术问题 ──▶ 调用 interrupt_handler 分类处理
```

## 7. 项目完成与归档

```
所有任务完成
    │
    ▼
项目总监汇总成果
    │
    ├─ 输出文件整理到 works/{project_id}/output/
    ├─ 生成项目总结报告
    └─ 在群中向用户交付最终成果
    │
    ▼
用户确认完成
    │
    ▼
项目总监发起归档：
  ├─ 更新 .meta.json 状态为 completed
  ├─ 归档各角色记忆（role_learner 异步触发）
  ├─ 清理临时 Agent 状态
  └─ 可选：保留 Telegram 群作为只读记录 / 或解散
    │
    ▼
实施角色（worker）处理：
  ├─ 释放到全局角色池（默认）──▶ 可被其他项目复用
  ├─ 归档（Soul 保留，不活跃）──▶ 保留项目关联，不可复用
  └─ 删除（admin 权限）──▶ 进入角色删除流程
    │
    ▼
项目总监角色可选择：
  ├─ 保留（用于后续维护）
  └─ 归档（Soul 保留，不再活跃）
    │
    ▼
项目归档完成
```

**角色复用规则**：
- 默认情况下，项目中的 worker 角色归档后释放到全局角色池
- 技能/洞察保留在角色目录中，跨项目继承
- 项目特定的上下文（如项目记忆）随项目归档，不进入全局池
- 用户可显式指定某个 worker 随项目一起归档（不进入全局池）
```

## 8. 项目状态机

```
planning ──▶ active ──▶ reviewing ──▶ completed
    │           │            │
    │           ├──▶ paused ─┘
    │           │
    │           └──▶ blocked ──▶ active（恢复）
    │
    └──▶ cancelled
```

| 状态 | 说明 |
|------|------|
| planning | suri 创建项目组，生成项目总监 Soul，用户确认 |
| active | 项目总监调度执行，用户在群中交互 |
| paused | 项目暂停，可恢复 |
| blocked | 受阻，等待 suri 或用户决策 |
| reviewing | 项目总监汇总成果，用户审核 |
| completed | 项目完成，归档 |
| cancelled | 项目取消 |

## 9. 跨项目知识复用

```
新项目启动
    │
    ▼
suri 查询历史项目（memory_service）
    │
    ├── 发现相似项目
    │       │
    │       ▼
    │   提取相关经验
    │       ├─ 成功模式
    │       ├─ 常见问题
    │       └─ 优化建议
    │       │
    │       ▼
    │   注入新项目规划参考
    │       └─ 项目总监 Soul 生成时参考历史经验
    │
    └── 无相似项目 ──▶ 从零规划
```
