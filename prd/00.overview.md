## 1. 项目定位

Suri 是一个**多角色 AI Agent 协同平台**，采用"董事长-CEO-职能部门"的组织隐喻：

- **用户** = 董事长（只与 suri 对话）
- **suri** = CEO（唯一调度入口，需求分析+任务分解+结果汇总）
- **部门负责人** = 总监（二次分解，指派执行角色）
- **执行角色** = 职能部门（调用工具/技能完成任务）
- **suri_hr** = 行政+人力（角色/部门/技能管理）
- **suri_dev** = 技术维护（主程序代码+工具开发）
- **suri_review** = 质量审核（文件变更审查）
- **suri_stats** = 数据分析（统计报告+监控预警）

## 3. 术语表

| 术语 | 定义 | 物理位置 |
|------|------|---------|
| **Soul** | 角色定义文件（YAML frontmatter + 能力描述） | `group/<dept>/<role>/<role>.md` |
| **Tool** | 平台公共通用能力（文件读写、数据库查询等） | `suri-agent/tools/<tool_id>/` |
| **Skill** | 角色私有能力模块（业务流程+脚本+资源） | `group/<dept>/<role>/skills/<skill_id>/` |
| **Insight** | 角色自学习提取的经验条目 | `group/<dept>/<role>/memories/insights/*.md` |
| **Agent** | 任务执行实例（含步骤、状态、上下文） | 内存 + `suri-agent/state/agents.db` |
| **TaskStep** | Agent 内的原子执行步骤 | 内存 + SQLite |
| **投影** | 将角色内部通信同步展示到 Telegram 群 | `access/projection.py` |
| **开发态** | 人类开发者在 IDE 中编写框架代码 | `suri-agent/` |
| **运行态** | Agent 在框架内自主执行业务 | `group/` + 运行时内存 |

## 4. 核心原则（不可违反）

### 4.1 空间分离
- `group/`（角色空间）与 `suri-agent/`（主程序空间）严格分离
- 角色只能通过 `ToolService` / `MemoryService` 等基础设施代理操作文件
- `suri_dev`（maintainer 类型）管理 `suri-agent/`；`suri_hr`（admin 类型）管理 `group/`

### 4.2 唯一入口
- 用户**只与 suri 对话**，其他角色不得直接向用户发送未经 suri 转述的消息
- 所有面向用户的输出必须经过 suri 汇总

### 4.3 用户确认机制
- 所有部门/角色/技能的创建必须经过用户多轮对话确认
- `CreationDialog` 状态机管理 3 种创建流程

### 4.4 文档同步
- 代码变更必须同步更新文档（AGENTS.md #1 绝对规则）
- `DocSyncRule` 引擎检测"代码已更新但文档未更新"违规项

### 4.5 测试同步
- 任何新功能或 Bug 修复必须配套测试用例
- 回归测试必须在修复前复现 Bug、修复后通过
