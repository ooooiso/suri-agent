# suri-agent 框架重构方案 V2

> 基于 suri 老项目（V4.0）能力差距分析，重构主程序插件并补齐缺失的执行层插件。
>
> **核心目标**：从"极简框架"升级为"可执行框架"——角色不仅能调用插件，还能通过执行层插件完成复杂任务的分解、调度、跟踪和通信。

## 二、重构方案概述

### 2.1 插件总数：13 → 20

新增 **7 个插件**，重构 **1 个插件**（suri_core）。

### 2.2 新增插件清单

| # | 插件名 | 来源 | 职责 | 优先级 |
|---|--------|------|------|--------|
| 14 | **task_scheduler** | 老项目 scheduler | 任务优先级队列、并发控制、超时重试、LLM 响应等待 | **P0** |
| 15 | **task_planner** | 老项目 task_plan | 任务分解、依赖管理（DAG）、预设模板、LLM 辅助规划 | **P0** |

| 17 | **role_comm** | 老项目 message_bus + messenger + communication | 角色间点对点/广播消息、权限规则、持久化队列、留存策略 | **P0** |
| 18 | **interrupt_handler** | 老项目 interrupt_handler | 受阻原因分类、用户建议生成、升级通道 | **P1** |
| 19 | **upgrade_manager** | 老项目 upgrade_report + report_manager | 升级报告状态机、闭环检查、Finding/UpgradeReport 模型 | **P1** |
| 20 | **doc_sync** | 老项目 doc_sync + doc_watcher | 文件变更监控、LLM 生成文档更新建议、用户确认写入 | **P2** |

### 2.3 suri_core 重构

**当前问题**：EventRouter 定位模糊，与 EventBus 职责重叠。

**重构方向**：
- **保留**：EventBus（订阅匹配 + 事件分发）+ PluginManager（扫描/加载/生命周期）
- **合并**：EventRouter 作为 EventBus 的内部分发逻辑，不独立暴露
- **明确边界**：suri_core 只做"消息路由"和"插件生命周期管理"，不做任何任务执行逻辑

```
重构前：                          重构后：
suri_core                         suri_core
├── EventBus                      ├── EventBus（含内部分发逻辑）
├── PluginManager                 ├── PluginManager
└── EventRouter  ← 冗余            └── （EventRouter 合并入 EventBus）
```

## 三、新增插件详细设计

### 3.1 task_scheduler — 任务调度器

**定位**：任务执行调度中心。管理任务的优先级队列、并发控制、超时重试。被角色调用或订阅 task 事件。

**核心能力**：

```python
class TaskScheduler:
    """任务调度器"""
    
    # 1. 优先级队列
    _queue: asyncio.PriorityQueue  # (priority, timestamp, task_id)
    
    # 2. 并发控制
    _semaphore: asyncio.Semaphore(max_concurrent=10)
    
    # 3. LLM 响应等待
    _pending_llm: Dict[str, asyncio.Event]
    
    # 4. 任务状态跟踪（运行时）
    _running_tasks: Dict[str, asyncio.Task]
```

**关键方法**：

| 方法 | 说明 |
|------|------|
| `schedule(task_id, priority, executor_role, timeout)` | 将任务放入队列 |
| `cancel(task_id)` | 取消任务 |
| `pause(task_id)` | 暂停任务 |
| `resume(task_id)` | 恢复任务 |
| `_process_task(task_id)` | 内部执行：获取 semaphore → 调用角色 → 等待完成 |
| `_wait_for_llm(task_id, timeout)` | 等待 LLM 响应（asyncio.Event） |

**订阅事件**：
- `task.created` → 自动调度（如果配置了自动调度）
- `task.priority_changed` → 重新排序
- `task.cancel_requested` → 取消任务

**发布事件**：
- `task.started` → 任务开始执行
- `task.completed` / `task.failed` / `task.timeout` → 执行结果
- `task.cancelled` → 任务被取消

**配置项**：
```yaml
task_scheduler:
  max_concurrent: 10          # 最大并发任务数
  default_timeout: 300        # 默认超时（秒）
  retry_policy:
    max_retries: 3
    intervals: [0, 30, 120]   # 首次立即重试，第二次 30s，第三次 120s
  llm_wait_timeout: 60        # LLM 响应等待超时
  priority_weights:           # 优先级权重
    CRITICAL: 0
    HIGH: 1
    NORMAL: 2
    LOW: 3
```

**依赖**：suri_core、llm_gateway（等待响应）、agent_registry（更新状态）

### 3.2 task_planner — 任务规划器

**定位**：任务分解与依赖管理。被角色调用，将复杂任务分解为可执行的步骤序列。

**核心能力**：

```python
class TaskPlanner:
    """任务规划器"""
    
    # 预设任务模板
    TASK_TEMPLATES = {
        "code": {
            "steps": ["理解需求", "识别依赖", "设计", "编码", "自测", "交付"],
            "default_role": "suri_dev"
        },
        "review": {
            "steps": ["收集变更", "逐文件审查", "影响分析", "出具报告"],
            "default_role": "suri_review"
        },
        "statistics": {
            "steps": ["数据抽取", "清洗", "计算聚合", "可视化"],
            "default_role": "suri_data"
        },
        "role_creation": {
            "steps": ["分析需求", "设计能力矩阵", "生成 Soul", "创建目录", "通知 suri"],
            "default_role": "suri_hr"
        }
    }
```

**关键方法**：

| 方法 | 说明 |
|------|------|
| `plan(task_text, context)` | 任务规划入口 |
| `_rule_based_plan(task_text)` | 基于模板规则规划（快速路径） |
| `_llm_plan(task_text, context)` | 基于 LLM 的规划（智能路径） |
| `_build_dag(steps)` | 构建步骤依赖图 |
| `get_ready_steps(plan_id)` | 获取当前可执行的步骤（依赖已满足） |

**数据模型**：

```python
@dataclass
class TaskPlan:
    plan_id: str
    task_name: str
    steps: List[TaskStep]
    involved_roles: List[str]
    dependencies: List[str]  # 步骤间依赖
    estimated_total_time: Optional[int] = None

@dataclass
class TaskStep:
    step_id: str
    description: str
    assignee: str          # 负责角色
    status: str            # pending | in_progress | completed | blocked
    depends_on: List[str]  # 前置步骤 ID
    estimated_time: Optional[int] = None
    result: Optional[str] = None
```

**订阅事件**：
- `task.plan_requested` → 生成任务规划

**发布事件**：
- `task.planned` → 规划完成
- `task.step_ready` → 步骤可执行
- `task.plan_updated` → 规划变更（如步骤被阻塞）

**依赖**：suri_core、llm_gateway、role_manager

### 3.3 agent_registry — Agent 注册表

**定位**：Agent 生命周期管理。创建/销毁 Agent，跟踪状态，支持父子关系。

**核心能力**：

```python
class AgentRegistry:
    """Agent 注册表"""
    
    # 运行时内存
    _agents: Dict[str, Agent]           # agent_id → Agent
    _contexts: Dict[str, AgentContext]  # agent_id → AgentContext
    
    # 持久化
    _task_state: TaskStateService  # SQLite
```

**关键方法**：

| 方法 | 说明 |
|------|------|
| `create_agent(task_text, role_id, user_id)` | 创建 Agent |
| `create_sub_agent(parent_id, subtask, role_id, user_id)` | 创建子 Agent |
| `get_agent(agent_id)` | 获取 Agent |
| `update_step(agent_id, step_id, status, result)` | 更新步骤状态 |
| `block_agent(agent_id, reason)` | 标记 Agent 受阻 |
| `complete_agent(agent_id)` | 标记 Agent 完成 |
| `get_progress(agent_id)` → str | 获取进度（如 "2/4"） |
| `cleanup_old_agents(max_age_hours=24)` | 清理已完成的老 Agent |

**数据模型**：

```python
@dataclass
class Agent:
    agent_id: str
    task_id: str
    task_name: str
    parent_agent_id: Optional[str]   # 父子关系
    role_id: str
    status: str                      # planning | running | paused | completed | blocked | cancelled
    steps: List[TaskStep]
    user_id: str
    created_at: str
    updated_at: str
    
    @property
    def progress(self) -> str:
        completed = sum(1 for s in self.steps if s.status == "completed")
        return f"{completed}/{len(self.steps)}"
    
    @property
    def current_step(self) -> Optional[TaskStep]:
        for s in self.steps:
            if s.status == "in_progress":
                return s
        for s in self.steps:
            if s.status == "pending":
                return s
        return None

class AgentContext:
    """Agent 独立上下文"""
    def __init__(self, agent_id: str):
        self.agent_id = agent_id
        self._messages: List[Dict] = []
    
    def add_message(self, role: str, content: str):
        self._messages.append({"role": role, "content": content, "timestamp": datetime.now().isoformat()})
    
    def build_chat_messages(self, role_id: str, task_hint: str = "") -> List[Dict]:
        # system prompt + 任务提示 + 历史消息
        ...
```

**订阅事件**：
- `agent.create_requested` → 创建 Agent
- `agent.step_update` → 更新步骤

**发布事件**：
- `agent.created` → Agent 创建完成
- `agent.status_changed` → 状态变更
- `agent.completed` → Agent 完成
- `agent.blocked` → Agent 受阻

**依赖**：suri_core、memory_service（持久化）

### 3.4 role_comm — 角色通信

**定位**：角色间消息通信服务。提供点对点、广播、持久化消息队列。

**核心能力**：

```python
class RoleComm:
    """角色通信插件"""
    
    # SQLite 持久化消息队列
    _message_bus: MessageBus
    
    # 通信规则引擎
    _rules: CommunicationRules
```

**关键方法**：

| 方法 | 说明 |
|------|------|
| `send(sender, receiver, msg_type, content, task_id)` | 发送点对点消息 |
| `broadcast(sender, content, msg_type)` | 广播消息 |
| `request_help(sender, content, task_id)` | 向 suri 求助 |
| `consume(receiver, limit=50)` | 消费消息（原子标记已消费） |
| `peek(receiver, limit=50)` | 查看消息（不消费） |
| `get_unread_count(receiver)` | 未读消息数 |

**消息格式**：

```python
@dataclass
class RoleMessage:
    msg_id: str
    sender: str
    receiver: str          # 或 "broadcast"
    msg_type: str          # task | approval | notify | escalation | status_update | request_help | completion
    content: str
    task_id: Optional[str]
    agent_id: Optional[str]
    priority: str          # high | normal | low
    timestamp: str
```

**通信权限规则**：

| 场景 | 是否允许 | 通道 |
|------|---------|------|
| 自己 → 自己 | ✅ | self |
| 同部门 | ✅ | department_group |
| 角色 ↔ suri | ✅ | private |
| 跨部门总监级 | ✅ | director_private_with_cc |
| 跨部门普通成员 | ❌ | - |
| 异常/审批/回流 | ✅（抄送 suri）| security_to_suri_to_user |

> 注：部门概念已移除，权限规则简化为"同工作组允许，跨工作组需审批"。

**留存策略**：

| 消息类型 | 留存天数 |
|----------|---------|
| approval | 90 |
| task | 30 |
| notify | 30 |
| escalation | 90 |

**订阅事件**：
- `role.message` → 接收消息

**发布事件**：
- `role.message_received` → 消息到达
- `role.message_delivered` → 消息已投递

**依赖**：suri_core

### 3.5 interrupt_handler — 中断处理器

**定位**：任务执行受阻时的系统级处理。分类受阻原因，生成用户建议，提供升级通道。

**核心能力**：

```python
class InterruptHandler:
    """中断处理器"""
    
    def handle(self, agent_id: str, block_reason: str) -> InterruptResult:
        reason_type = self._classify_reason(block_reason)
        return self._handlers[reason_type](agent_id, block_reason)
```

**受阻原因分类**：

| 类型 | 关键词 | 处理策略 |
|------|--------|---------|
| `missing_tool` | 缺少工具/接口/功能 | 建议创建工具或提交工具开发需求 |
| `knowledge_gap` | 知识不足/不会/不了解 | 建议搜索或询问用户 |
| `permission_denied` | 权限不足/拒绝访问 | 建议申请权限或降级执行 |
| `dependency_failed` | 依赖失败/上游错误 | 建议重试、跳过或升级 |

**中断结果**：

```python
@dataclass
class InterruptResult:
    handled: bool
    action: str            # wait | escalate | cancel | auto_resolve
    suggestion: str        # 给用户的中文建议
    new_agent_id: Optional[str]  # 升级后创建的新 Agent
    reason: str            # 原因类型
```

**升级通道**：

| 升级目标 | 场景 | 方式 |
|----------|------|------|
| `escalate_to_dev()` | 缺少工具/技术问题 | 向 suri_dev 发送 escalation 消息 |
| `escalate_to_hr()` | 需要新角色/权限问题 | 向 suri_hr 发送 escalation 消息 |
| `escalate_to_user()` | 需要用户决策 | 通过 access 向用户呈现选项 |

**订阅事件**：
- `agent.blocked` → 处理中断
- `task.failed` → 分析是否需要中断处理

**发布事件**：
- `interrupt.handled` → 中断已处理
- `interrupt.escalated` → 已升级
- `interrupt.user_decision_needed` → 需要用户决策

**依赖**：suri_core、role_comm、agent_registry

### 3.6 upgrade_manager — 升级报告管理器

**定位**：管理升级报告的完整生命周期。提供状态机、闭环检查、结构化数据模型。

**核心能力**：

```python
class UpgradeManager:
    """升级报告管理器"""
    
    # 报告存储
    _storage_path: Path  # ~/.suri/data/upgrade_reports/
```

**报告状态机**：

```
PENDING ──→ SUBMITTED ──→ ACKNOWLEDGED ──→ DISPATCHED ──→ IMPLEMENTED
   │            │              │                 │
   └────────────┴──────────────┴─────────────────┴──→ REJECTED
   │
   └──→ DEFERRED
```

**数据模型**：

```python
class ReportStatus(str, Enum):
    PENDING = "pending"           # 刚生成，未提交
    SUBMITTED = "submitted"       # 已提交给 suri
    ACKNOWLEDGED = "acknowledged" # suri 已确认收到
    DISPATCHED = "dispatched"     # 已分发给相关方
    IMPLEMENTED = "implemented"   # 已实施
    REJECTED = "rejected"         # 被拒绝
    DEFERRED = "deferred"         # 延期

class Priority(str, Enum):
    CRITICAL = "critical"
    HIGH = "high"
    MEDIUM = "medium"
    LOW = "low"

class FindingCategory(str, Enum):
    PERFORMANCE = "performance"
    ACCURACY = "accuracy"
    USABILITY = "usability"
    SECURITY = "security"
    INTEGRATION = "integration"
    DESIGN = "design"
    RELIABILITY = "reliability"

@dataclass
class Finding:
    finding_id: str
    category: FindingCategory
    description: str
    evidence: str
    suggestion: str
    estimated_impact: str
    confidence: float

@dataclass
class UpgradeReport:
    report_id: str
    module_id: str           # 插件/角色/框架 ID
    module_type: str         # plugin | tool | role | framework
    findings: List[Finding]
    summary: str
    priority: Priority
    status: ReportStatus
    submitted_to: str = "suri"
    created_at: str = ""
    updated_at: str = ""

@dataclass
class FrameworkImprovementReport:
    report_id: str
    findings: List[Finding]
    analysis_scope: str
    common_issues: List[str]
    framework_risks: List[str]
    estimated_change_difficulty: str
```

**关键方法**：

| 方法 | 说明 |
|------|------|
| `save_report(report)` | 保存报告到文件系统 |

### 6.1 process.md — 工作流程定义

定义系统的标准工作流程：

| 流程 | 场景 | 步骤 |
|------|------|------|
| **标准任务流** | 单部门内任务 | 用户 → ruri → 角色匹配 → task_planner 分解 → task_scheduler 调度 → agent_registry 跟踪 → 执行 → 汇总 → 交付 |
| **多角色协作流** | 复杂任务需多角色 | ruri 确定涉及角色 → task_planner 生成协作计划 → task_scheduler 并行调度 → 各角色执行 → 每 N 分钟同步进度 → ruri 汇总 |
| **异常处理流** | 任务执行失败 | 失败计数 → 重试（退避 0s/30s/120s）→ 超次 → interrupt_handler 分类 → 升级/取消 |
| **用户决策回路** | 需要用户确认 | 中断 → 整理上下文 + 2~3 方案 → 用户决策 → 继续执行 |
| **能力缺口流** | 现有角色无法处理 | 遍历能力索引 → 无匹配 → 向用户展示缺口 → 用户确认 → role_manager 创建新角色 → 重新调度 |
| **自优化上报流** | 系统自我改进 | PluginSelfLearning 分析 → 生成 UpgradeReport → upgrade_manager 保存 → suri 角色汇总 → 用户确认 → 实施 |

### 6.2 rules.md — 系统规则

定义系统运行规则：

| 规则 | 说明 |
|------|------|
| **通信规则** | 同工作组自由通信，跨工作组需审批；所有消息持久化 30~90 天 |
| **安全规则** | 插件加载前 AST 扫描；文件变更需审批令牌；Soul 文件只读 |

## 七、实施建议

### 7.1 优先级排序

| 阶段 | 插件 | 理由 |
|------|------|------|
| **P0（必须先做）** | task_scheduler + task_planner + agent_registry | 没有这三个，系统无法执行复杂任务 |
| **P0（必须先做）** | role_comm | 没有角色通信，多角色无法协作 |
| **P1（尽快补齐）** | interrupt_handler | 系统鲁棒性关键 |
| **P1（尽快补齐）** | upgrade_manager + PluginSelfLearning 增强 | 元学习闭环 |
| **P2（可以延后）** | doc_sync | 开发体验优化 |
| **P2（可以延后）** | 投影服务 | 观察窗口，非核心 |

### 7.2 代码复用策略

| 插件 | 老项目代码 | 复用度 | 适配工作 |
|------|-----------|--------|---------|
| task_scheduler | `core/scheduler/__init__.py` | 80% | 适配为插件接口（事件驱动） |
| task_planner | `core/task_plan.py` | 85% | 适配路径（group/ → role/），移除部门依赖 |
| agent_registry | `core/agent_registry.py` + `core/task_state.py` | 80% | 合并两个文件，适配插件接口 |
| role_comm | `core/message_bus.py` + `role/messenger.py` + `rules/communication.py` | 75% | 简化权限规则（移除部门），适配事件总线 |
| interrupt_handler | `core/interrupt_handler.py` | 90% | 几乎直接复用 |
| upgrade_manager | `meta/upgrade_report.py` | 85% | 适配为插件，添加事件接口 |
| PluginSelfLearning | `meta/module_learner.py` | 80% | 从独立类改为共享模块基类 |

### 7.3 suri_core 重构步骤

1. **移除 EventRouter 独立概念**：将 EventRouter 的投递逻辑合并到 EventBus 内部
2. **明确边界**：suri_core 只暴露 EventBus + PluginManager 两个核心组件
3. **事件订阅标准化**：所有插件通过 EventBus 订阅事件，不再直接依赖 suri_core 的方法调用
4. **健康检查增强**：suri_core 升级后的冒烟测试包含 EventBus 分发 + PluginManager 加载流程

## 八、总结

### 重构前后对比

| 维度 | 重构前（13 插件） | 重构后（20 插件） |
|------|-----------------|-----------------|
| **任务执行** | 无调度，角色自己决定怎么执行 | task_scheduler 统一调度，task_planner 自动分解 |
| **多 Agent** | 无 Agent 概念 | agent_registry 支持子 Agent + 状态跟踪 |
| **角色通信** | 裸 EventBus（只有发布-订阅） | role_comm 支持点对点/广播/权限/持久化 |
| **中断处理** | 无 | interrupt_handler 分类受阻 + 升级通道 |
| **元学习闭环** | 流程有，缺状态机 | upgrade_manager 完整状态机 + 自动检查 |
| **文档同步** | 无 | doc_sync 自动保持代码-文档一致 |
| **架构层次** | 基础设施 + 能力层 | 基础设施 + 执行层 + 能力层 |

### 核心改进

1. **执行层补齐**：系统从"只能做简单插件调用"升级为"能分解、调度、跟踪复杂多角色任务"
2. **角色真正协作**：通过 role_comm 实现角色间的结构化通信，不只是事件广播
3. **元学习闭环**：upgrade_manager 让"发现问题→生成报告→跟踪实施"形成完整闭环
4. **老代码高复用**：7 个新插件中，75~90% 的逻辑可直接从老项目复用，主要工作是适配插件接口和移除部门依赖
