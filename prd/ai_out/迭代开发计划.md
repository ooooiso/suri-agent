# Suri Agent 迭代开发计划

> 基于全部 PRD 文档制定的五迭代开发规划。严格遵循 PRD 定义，每个迭代结束时对照 PRD 回归验证。

---

## 迭代原则

1. **逐层递进**：每个迭代在上一个迭代可运行的基础上叠加新能力
2. **PRD 驱动**：所有开发严格参考 PRD，实现与文档不一致时优先回归 PRD
3. **插件化**：所有功能以插件形式实现，无硬编码耦合
4. **零依赖**：仅使用 Python 标准库 + asyncio + SQLite
5. **测试闭环**：每个迭代结束时必须有完整测试覆盖本迭代所有链路

---

## 迭代 1：终端与 Telegram 对话（MVP）

### 目标
实现用户通过终端和 Telegram 与 suri 对话的最小可用系统。核心是让 agent 连接终端和第三方会话。

### 包含插件（8 个）

| 插件 | 必须/可选 | 说明 |
|------|----------|------|
| **suri_core** | 必须 | EventBus + PluginManager，系统启动基础 |
| **access** | 必须 | CLI 通道（`suri` 命令进入会话）+ Telegram Bot |
| **config_service** | 必须 | 模型配置、API Key、Telegram Token 存储与热重载 |
| **llm_gateway** | 必须 | 5 个国内模型接入、版本切换、异常切换 |
| **log_service** | 必须 | 基础日志记录 |
| **security_service** | 必须（简化） | 基础文件权限检查、配置敏感信息保护 |
| **role_manager** | 必须（简化） | 创建核心角色 suri、管理 Soul 文件 |
| **memory_service** | 可选 | 基础会话记忆存储（如迭代时间允许） |

### 不包含（后续迭代）
task_planner、agent_registry、task_scheduler（复杂调度）、role_comm、interrupt_handler、role_learner、upgrade_manager、doc_sync、cron_service、hooks_service、test_framework（作为外部测试工具使用）、mcp_framework

### 核心功能链路

```
用户输入 "suri"
    │
    ▼
main.py（<20 行）
    │
    ▼
SuriCorePlugin.bootstrap() → 创建 EventBus → 创建 PluginManager
    │
    ▼
自注册 suri_core → 按依赖顺序加载其他插件
    │
    ▼
access CLI 通道启动 → 进入交互模式
    │
    ├── 首次运行 → 配置向导
    │   ├─ 模型选择（5 个国内模型列表）
    │   ├─ API Key 输入（加密存储）
    │   ├─ 模型版本切换（如 flash/pro）
    │   └─ Telegram Bot Token（可选，可跳过）
    │
    └── 非首次运行 → 直接进入会话
    │
    ▼
用户输入消息
    │
    ▼
access 发布 user.input 事件
    │
    ▼
suri 角色（由 role_manager 创建）订阅 user.input
    │
    ▼
suri 调用 llm_gateway 生成回复
    │
    ▼
回复通过 access 返回终端（或 Telegram）
```

### 支持的自然语言命令

| 命令意图 | 示例输入 | 处理 |
|----------|----------|------|
| 切换模型 | "切换成文心一言" | config_service 更新配置 → llm_gateway 切换 |
| 新增模型 | "添加通义千问" | 进入模型配置流程 |
| 配置 Telegram | "帮我配置 Telegram" | 进入 Telegram 配置流程 |
| 查看配置 | "当前用的是什么模型" | config_service 读取并返回 |
| 普通对话 | "你好 suri" | 直接通过 llm_gateway 生成回复 |

### 5 个国内模型（首批支持）

| 厂商 | 模型 | 版本示例 |
|------|------|----------|
| 百度 | 文心一言 | ERNIE-Bot / ERNIE-Bot-4 |
| 阿里 | 通义千问 | qwen-turbo / qwen-plus / qwen-max |
| 智谱 | ChatGLM | glm-4-flash / glm-4-plus |
| 月之暗面 | Kimi | kimi-moonshot-v1-8k / kimi-moonshot-v1-32k |
| DeepSeek | DeepSeek | deepseek-chat / deepseek-coder |

### 模型异常处理

- 网络超时/断开 → llm_gateway 返回 llm.error → access 提示用户
- 用户在终端输入 "切换模型" 或 "换成 xxx" → 直接触发模型切换流程
- API Key 失效 → 提示重新输入

### Telegram 链路

```
用户自然语言 "配置 Telegram"
    │
    ▼
suri 理解意图 → 请求 Bot Token
    │
    ▼
用户输入 Token → config_service 保存
    │
    ▼
access Telegram 通道启动 → 轮询 Bot API
    │
    ▼
用户向 Bot 发消息 → Telegram 通道接收
    │
    ▼
发布 user.input（channel=telegram）→ suri 处理
    │
    ▼
回复通过 Telegram Bot API 发回用户
```

### 开发顺序

1. **Day 1-2**：suri_core（EventBus + PluginManager）+ main.py
2. **Day 3**：config_service + log_service + security_service（简化）
3. **Day 4-5**：llm_gateway（模型接入、版本切换、异常处理）
4. **Day 6-7**：access CLI 通道（配置向导、会话交互）
5. **Day 8-9**：role_manager（创建 suri 角色）+ access Telegram 通道
6. **Day 10**：整合测试 + 回归 PRD

### 测试要求

- [ ] 终端输入 `suri` 能正常启动并进入会话
- [ ] 首次运行配置向导能完整走完（模型+API Key+Telegram）
- [ ] 5 个模型都能正常对话
- [ ] 模型版本切换正常（如 flash → pro）
- [ ] 模型异常时能自然语言切换
- [ ] Telegram 配置链路完整（配置→对话→回复）
- [ ] 配置可跳过，后续自然语言唤醒配置
- [ ] 日志正确记录所有事件
- [ ] 配置热重载生效

## 迭代 2：任务执行与开发辅助

### 目标
让 suri 能够执行复杂任务，并通过连接 LLM 辅助后续迭代的开发规划。

### 新增插件（4 个）

| 插件 | 说明 |
|------|------|
| **task_planner** | 任务分解、DAG 依赖管理 |
| **agent_registry** | Agent 生命周期、父子关系、进度跟踪 |
| **task_scheduler** | 优先级队列、并发控制、超时重试 |
| **interrupt_handler** | 受阻处理、用户决策（简化） |

### 核心能力

1. **复杂任务执行**
   - 用户输入复杂需求 → suri 调用 task_planner 分解 → task_scheduler 调度 → agent_registry 跟踪 → 返回结果

2. **开发辅助**
   - suri 通过 LLM 分析 PRD 文档，辅助规划后续迭代
   - suri 可以阅读代码、提出建议、生成伪代码
   - 不直接修改代码，输出到终端供用户确认

3. **测试框架 test_framework**
   - 将迭代 1 的手工测试固化为自动化测试
   - 覆盖所有基础插件

### 不包含
- 多角色协作（迭代 3）
- 角色学习（迭代 4）
- 工具调用（迭代 5）

## 迭代 3：角色协作与项目管理

### 目标
支持多角色协作和项目级工作流，实现 Telegram 项目群。

### 新增插件（2 个）+ 角色能力

| 插件/角色 | 说明 |
|-----------|------|
| **role_comm** | 角色间点对点/广播通信 |
| **Project Director** 角色 | 自动创建，负责项目调度 |
| **work_flow** 流程 | 项目创建→执行→归档完整流程 |

### 核心能力

1. **项目创建**
   - 用户通过 Telegram 创建项目 → suri 建议项目群 → 创建 works/{project_id}/
   - 自动创建 Project Director 角色

2. **多角色协作**
   - Project Director 分解任务 → 调度多个 worker 角色
   - 角色间通过 role_comm 通信
   - 进度在项目群中播报

3. **缺失角色处理**
   - Project Director 发现缺少能力 → 向 suri 报告 → 用户简化创建角色

## 迭代 4：学习与自进化

### 目标
角色能够从经验中学习，系统能够自我分析并提出优化建议。

### 新增插件（3 个）

| 插件 | 说明 |
|------|------|
| **role_learner** | 角色经验提取、洞察生成、技能检测 |
| **upgrade_manager** | 升级报告状态机、闭环检查 |
| **doc_sync** | 代码变更监控、文档更新建议 |

### 核心能力

1. **角色学习**
   - 任务完成后异步分析记忆
   - 生成洞察，注入后续任务上下文
   - 检测重复工具模式，建议技能

2. **全局分析（ProgramLearner）**
   - suri 定期请求全局分析
   - 生成系统优化报告
   - 进入统一升级流程（用户确认后执行）

3. **文档同步**
   - 代码变更后自动生成文档更新建议
   - 用户确认后应用

---

## 迭代 5：工具扩展与运维完善

### 目标
接入工具框架，完善运维能力，系统达到生产可用。

### 新增插件（3 个）

| 插件 | 说明 |
|------|------|
| **mcp_framework** | MCP 工具服务框架、内置工具注册 |
| **cron_service** | 定时任务调度 |
| **hooks_service** | 事件钩子拦截扩展 |

### 核心能力

1. **工具调用**
   - 文件读写、网络搜索、命令执行（受安全沙箱限制）
   - 工具注册、发现、权限控制

2. **定时任务**
   - 定时触发事件
   - 定期维护、报告生成

3. **事件钩子**
   - 拦截文件变更、任务事件
   - 扩展点供自定义逻辑

4. **运维完善**
   - 备份/恢复自动化
   - 监控指标、告警规则
   - 插件热重载
   - 数据库迁移工具

## 风险与回退策略

| 风险 | 应对 |
|------|------|
| LLM API 稳定性差 | llm_gateway 实现多模型降级切换 |
| 插件循环依赖 | 严格按 PRD 依赖关系加载，事件驱动解耦 |
| 配置丢失 | config_service 自动备份，首次运行向导可重新配置 |
| 迭代 1 范围膨胀 | 坚持 MVP，非核心功能果断推迟到后续迭代 |
