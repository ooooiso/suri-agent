# 核心流程分析

> 基于用户描述的多 Agent 智能体协同架构，梳理完整链路。
> 本文档覆盖：角色定位、完整链路、群组结构、索引清单规范、模板自动更新。

---

## 一、核心角色定位

### suri — 核心调度（唯一有修改权限的角色）

suri 的技能：

1. **创建角色** — 根据用户需求创建新角色，生成 Soul 草案
2. **维护主程序** — 修改代码、新增插件、更新配置、创建工具
3. **任务调度** — 在中台群组中协调各角色
4. **升级自身** — 自己不会时请求用户允许增加新技能
5. **创建项目** — 为大型任务创建项目空间
6. **创建机器人** — 为角色绑定 Telegram 等渠道

**suri 拥有修改一切的能力**，其他角色没有。

### 项目经理 — 项目调度

- 大型任务时由 suri 创建
- 在独立项目群组中与用户直接对话
- 分析需求，决定需要哪些角色
- 缺角色/缺工具 → 向 suri 中台申请，suri 转发用户确认

### 普通角色 — 业务执行者

- 每个角色是独立 Agent，有自己的 Soul、技能、记忆
- 角色根据 Soul 定义知道自己该做什么
- 角色通过调用 MCP 工具和大模型能力完成任务
- 角色可以自学习、自增技能（技能更新会反映在 Soul 中）
- 角色**没有**修改系统代码的权限

## 二、完整链路

### 链路 1：用户提出简单需求

```
用户: "请帮我写一份产品文档"
    │
    ▼
suri 判断: 这个需求在我的技能范围内吗？
    │
    ├─ 是 → 直接处理（仅限于创建角色/维护系统/调度任务）
    │
    └─ 否 →
        │
        ├─ 检查现有角色: 有没有角色 Soul 中定义了"文档撰写"技能？
        │   │
        │   ├─ 有 → 分配给该角色
        │   │   suri 在中台群组中调度该角色执行
        │   │
        │   └─ 没有 →
        │       │
        │       ▼
        │       suri 问用户: "当前没有角色能写产品文档，
        │       是否创建一个'文档撰写员'角色？我来生成 Soul 草案"
        │       │
        │       ▼
        │       用户确认 →
        │       │
        │       ▼
        │       suri 执行:
        │       1. 调用 role_manager 创建角色
        │       2. 生成 Soul 草案（让用户确认/优化）
        │       3. 创建 Telegram 机器人（或其他渠道）绑定该角色
        │       4. 创建"Suri 中台"群组（如果尚未创建）
        │       5. 将新角色加入中台群组
        │       │
        │       ▼
        │       新角色就绪后:
        │       - 角色根据 Soul 定义知道自己需要做什么
        │       - 角色查询可用 MCP 工具列表
        │       - 角色调用工具和大模型完成任务
        │       - 角色新增技能时更新自己的 Soul
        │       - 角色遇到问题在中台群组中向 suri 上报
        │       - suri 在中台群组中协调处理
```

### 链路 2：用户提出大型任务

```
用户: "帮我开发一个电商系统"
    │
    ▼
suri 判断: 这是一个大型任务，需要创建项目
    │
    ▼
suri 问用户: "这是一个大型项目，是否创建项目空间？
   请描述项目的作用和需求"
    │
    ▼
用户确认并描述需求 →
    │
    ▼
suri 执行:
    │
    ├─ 1. 在 works/ 中创建项目文件夹
    │   works/ecommerce/
    │   ├── .meta.json
    │   ├── prd.md（用户需求文档）
    │   ├── plan/
    │   ├── output/
    │   └── logs/
    │
    ├─ 2. 创建"项目经理"角色
    │   - 根据项目定义角色名称和 Soul
    │   - 创建 Telegram 机器人绑定项目经理
    │
    ├─ 3. 创建"项目名称"群组
    │   - 群组成员：用户 + 项目经理
    │   - 用户直接和项目经理对话
    │
    └─ 4. 项目经理就位后：
        │
        ▼
        项目经理分析需求 →
        │
        ├─ 判断需要哪些角色参与
        │   （前端开发、后端开发、UI设计、测试等）
        │
        ├─ 检查现有角色中哪些可用
        │
        ├─ 缺少的角色 → 向 suri 在中台群组中申请
        │   suri 转发给用户确认后创建
        │
        ├─ 需要新工具 → 向 suri 在中台群组中申请
        │   suri 转发给用户确认后创建
        │
        └─ 项目经理在项目群组中调度各角色完成任务
```

### 链路 3：项目经理核心调度链路

```
项目经理就位后，在项目群组中与用户直接对话
    │
    ▼
项目经理分析用户需求 →
    │
    ├─ 判断需要哪些角色参与
    │   （前端开发、后端开发、UI设计、测试等）
    │
    ├─ 检查现有角色清单（roles/README.md）
    │   │
    │   ├─ 已有合适角色 →
    │   │   向 suri 中台申请调度该角色加入项目
    │   │   suri 在中台群组中调度该角色
    │   │
    │   └─ 缺少的角色 →
    │       向 suri 在中台群组中申请创建新角色
    │       suri 转发给用户确认
    │       用户确认后 suri 创建角色并加入项目
    │
    ├─ 检查现有工具清单（plugins/README.md）
    │   │
    │   ├─ 已有合适工具 → 直接调用
    │   │
    │   └─ 缺少的工具 →
    │       向 suri 在中台群组中申请创建新工具
    │       suri 转发给用户确认
    │       用户确认后 suri 开发新工具
    │
    └─ 项目经理在项目群组中调度各角色协同完成任务
        │
        ├─ 角色完成任务 → 项目经理汇总结果 → 汇报用户
        ├─ 角色遇到问题 → 项目经理协调解决
        │   ├─ 需要其他角色 → 向 suri 中台申请调度
        │   └─ 需要新工具 → 向 suri 中台申请创建
        └─ 项目完成 → 项目经理归档 → suri 关闭项目
```

**项目经理 vs suri 的职责边界：**

| 场景 | 谁处理 | 说明 |
|------|--------|------|
| 用户提需求 | 项目经理 | 在项目群组中直接对话 |
| 需要角色 | 项目经理申请 → suri 执行 | suri 在中台创建/调度 |
| 需要工具 | 项目经理申请 → suri 执行 | suri 开发新工具 |
| 角色协同 | 项目经理调度 | 在项目群组中协调 |
| 问题上报 | 角色 → 项目经理 → suri | 逐级上报 |
| 项目归档 | 项目经理执行 | suri 确认关闭 |

### 链路 4：角色协同与问题上报

```
角色在执行任务时遇到问题
    │
    ├─ 需要其他角色协助 →
    │   在中台群组中向 suri 上报
    │   suri 协调其他角色加入
    │
    ├─ 需要新工具 →
    │   在中台群组中向 suri 申请
    │   suri 转发给用户确认
    │   用户确认后 suri 创建工具
    │
    └─ 需要新增技能 →
        角色自己新增技能
        技能更新反映在 Soul 中
        模板自动更新
```

## 三、群组结构与通信机制

> **核心原则：所有群组中的通信必须同步到所有角色。**
> 任何群组中的消息、决策、状态变更，都需要通过 EventBus 广播到所有角色，确保信息一致。

### 3.1 群组类型

#### Suri 中台群组

- **成员**：用户 + suri + 所有角色
- **用途**：
  - suri 调度各角色
  - 角色上报问题
  - 角色间协同申请
  - suri 转发用户确认
- **通信同步**：所有消息广播到全部角色

#### 项目群组

- **成员**：用户 + 项目经理
- **用途**：
  - 用户直接和项目经理对话
  - 项目经理分析需求、调度角色
  - 项目经理向 suri 中台申请资源
- **通信同步**：项目群组中的需求变更、角色调度、进度更新，同步到中台群组所有角色

### 3.2 通信同步机制

```
项目群组中用户说："需要增加一个数据库管理员"
    │
    ▼
项目经理收到消息
    │
    ├─ 向 suri 中台申请创建角色
    │
    ▼
suri 在中台群组中转发给用户确认
    │
    ▼
用户确认
    │
    ▼
suri 创建角色
    │
    ├─ 新角色加入中台群组 → 所有角色收到通知
    ├─ 新角色加入项目群组 → 项目经理和用户收到通知
    └─ 角色清单更新 → 所有角色同步最新清单
```

### 3.3 同步内容

| 同步事件 | 触发者 | 广播范围 | 说明 |
|---------|--------|---------|------|
| 角色创建/删除 | suri | 所有群组 | 角色清单变更 |
| 角色技能更新 | 角色自身 | 所有群组 | Soul 变更 |
| 任务分配 | suri/项目经理 | 所有群组 | 谁做什么 |
| 任务进度 | 执行角色 | 所有群组 | 当前状态 |
| 问题上报 | 任何角色 | 中台群组 | 需要协调 |
| 资源申请 | 项目经理 | 中台群组 | 缺角色/缺工具 |
| 用户决策 | 用户 | 所有群组 | 确认/拒绝/修改 |
| 工具新增/更新 | suri | 所有群组 | 工具清单变更 |
| 项目创建/关闭 | suri | 所有群组 | 项目状态变更 |

### 3.4 同步实现方式

```
群组消息
    │
    ├─ 通过 access 插件发送到对应渠道（Telegram/CLI）
    │
    ├─ 通过 EventBus 发布事件
    │   ├─ group.message（群组消息事件）
    │   ├─ role.status_changed（角色状态变更）
    │   ├─ task.assigned（任务分配）
    │   └─ resource.requested（资源申请）
    │
    └─ 所有订阅了对应事件的角色收到通知
        ├─ suri 收到所有事件
        ├─ 项目经理收到项目相关事件
        └─ 普通角色收到与自己相关的事件
```

## 四、索引清单规范

> 根目录下建立三个快速索引清单，方便 suri 调度时快速查询。
> 角色 + 插件 + 插件下的 MCP 工具能力（suri 开发工具，所有角色调用工具）。

### 4.1 角色清单（`roles/README.md`）

每个角色目录下包含 `soul.md`（角色定义）和 `meta.json`（元信息）。

**`meta.json` 规范：**

```json
{
  "role_id": "角色唯一标识",
  "nickname": "显示名称",
  "role_type": "core | worker | project_manager",
  "version": "1.0.0",
  "status": "active | paused | archived",
  "created_at": "创建时间",
  "updated_at": "更新时间",
  "channel": {
    "type": "telegram | cli",
    "bot_token": "机器人Token（如有）",
    "group_id": "所在群组ID（如有）"
  },
  "skills": ["技能1", "技能2"],
  "project_id": "所属项目ID（如有）",
  "parent_role": "父角色ID（如有）"
}
```

**角色类型：**

| 类型 | 说明 | 权限 |
|------|------|------|
| `core` | 核心调度（suri） | 修改一切 |
| `project_manager` | 项目经理 | 调度项目内角色 |
| `worker` | 普通执行者 | 调用工具和大模型 |

**快速查询命令：**

```bash
# 按技能查询
grep -r "skills:" roles/*/meta.json

# 按类型查询
grep -r '"role_type": "worker"' roles/*/meta.json

# 按状态查询
grep -r '"status": "active"' roles/*/meta.json
```

### 4.2 插件清单（`plugins/README.md`）

每个插件目录下包含 `manifest.json`（插件元信息）和 `plugin.py`（入口）。

**`manifest.json` 规范：**

```json
{
  "manifest_version": "1.0",
  "name": "插件ID",
  "version": "1.0.0",
  "entry_point": "plugin.py",
  "min_suri_version": "1.0.0",
  "type": "service | tool | capability | integration",
  "description": "插件功能描述",
  "permissions": ["权限列表"],
  "event_subscriptions": ["事件列表"],
  "fs_permissions": {
    "read": ["可读路径"],
    "write": ["可写路径"]
  },
  "runtime_mutable": true,
  "dependencies": ["依赖插件"],
  "config_schema": {}
}
```

**插件类型：**

| 类型 | 说明 | 示例 |
|------|------|------|
| `service` | 基础服务 | config_service, llm_gateway, log_service, security_service |
| `tool` | 工具能力 | code_tool |
| `capability` | 业务能力 | task_planner, task_scheduler, agent_registry, interrupt_handler |
| `integration` | 外部集成 | access |

**当前插件列表：**

| 插件 ID | 类型 | 描述 |
|---------|------|------|
| `access` | integration | 终端 CLI 与 Telegram Bot 访问通道 |
| `agent_registry` | capability | Agent 生命周期管理 |
| `code_tool` | tool | 代码读写执行工具 |
| `config_service` | service | 统一配置中心，支持热重载 |
| `interrupt_handler` | capability | 任务执行受阻时的系统级处理 |
| `llm_gateway` | service | 5 家国内大模型路由与多版本管理 |
| `log_service` | service | 分级日志、分类归档 |
| `role_manager` | service | 角色管理：核心角色 suri 的创建与 Soul 管理 |
| `security_service` | service | 安全沙箱与权限管控 |
| `task_planner` | capability | 任务分解引擎 |
| `task_scheduler` | capability | 任务调度中心 |

### 4.3 工具清单（`plugins/README.md` 中工具索引部分）

> 插件下的 MCP 工具能力。suri 开发工具，所有角色调用工具。

每个工具插件在 `manifest.json` 中定义其提供的工具能力：

```json
{
  "name": "code_tool",
  "type": "tool",
  "tools": [
    {
      "name": "read_file",
      "description": "读取文件内容",
      "params": ["path"]
    },
    {
      "name": "search_files",
      "description": "正则搜索文件",
      "params": ["path", "regex"]
    },
    {
      "name": "list_files",
      "description": "列出目录文件",
      "params": ["path"]
    }
  ]
}
```

**工具调用链路：**

```
角色需要读取文件
    │
    ├─ 查询工具清单 → 找到 code_tool.read_file
    ├─ 调用 code_tool 插件
    ├─ security_service 鉴权
    └─ 执行并返回结果
```

## 五、模板自动更新

### 需要自动更新的内容

| 模板 | 内容 | 更新方式 |
|------|------|---------|
| 角色技能清单 | 每个角色能做什么 | 角色新增技能时自动更新 Soul |
| 工具清单 | 有哪些 MCP 工具可用 | 工具注册时自动更新 |
| 插件清单 | 有哪些插件可用 | 插件加载时自动更新 |

### 更新流程

```
角色新增技能
    │
    ├─ 更新自己的 Soul 文件
    ├─ 发布技能更新事件
    │
    ▼
template_updater 接收事件
    ├─ 更新 tool_descriptions.yaml
    ├─ 通知其他角色刷新可用技能列表
    └─ 热更新生效
```

## 六、关键约束

1. **suri 拥有修改一切的能力**，其他角色没有
2. **角色根据 Soul 定义驱动**，suri 只关心每个角色的 Soul 是做什么的
3. **角色可以自学习、自增技能**，技能更新反映在 Soul 中
4. **所有角色必须加入中台群组**，由 suri 统一调度
5. **大型任务创建项目**，项目经理在项目群组中独立调度
6. **项目经理需要资源时向 suri 中台申请**，suri 转发用户确认
7. **普通角色主要调用 MCP 工具和大模型能力**，不修改系统
