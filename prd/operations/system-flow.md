# 系统工作流程定义

> 定义 suri-agent 的标准工作流程。suri 按 Soul 行事，在自己的技能范围内直接处理，范围外调度/升级。
所有流程由角色通过调用插件能力完成。

---

## 1. 用户请求处理流（核心流程）

**场景**：用户发起任何请求时的标准处理流程

```
用户: "请帮我写一份产品文档"
    │
    ▼
access 接收输入 → 发布 user.input 事件
    │
    ▼
suri 角色接收请求
    │
    ▼
┌──────────────────────────────────────────────────────────────┐
│ suri 判断：这个需求在我的技能范围内吗？                       │
│                                                              │
│ suri 是主人 Agent（role_type=core），职责包括：                │
│   1. 创建角色 — 当没有合适角色时创建新角色                    │
│   2. 维护主程序 — 修改代码、新增插件、更新配置                │
│   3. 任务调度 — 将任务分配给合适的角色                        │
│   4. 升级自身 — 当能力不足时申请增加新技能                    │
│   5. 业务处理 — 在自己的技能范围内直接处理需求                │
│                                                              │
│ 核心原则：suri 按 Soul.md 行事。技能范围内直接处理，           │
│ 范围外创建角色或调度。                                       │
└──────────────────┬───────────────────────────────────────────┘
                    │
     ┌──────────────┼──────────────┐
     │              │              │
     ▼              ▼              ▼
 在技能范围内     调度/维护任务   完全超出能力
     │              │              │
     ▼              ▼              ▼
 suri 直接      检查是否有合适    suri 问用户：
 调用工具处理    角色/维护能力    "是否允许我增加新技能？"
     │              │              │
     ▼       ┌──────┴──────┐      ├── 允许 → 升级自身
  完成处理    │             │      └── 拒绝 → 告知用户无法处理
          有匹配资源    无匹配资源
             │             │
             ▼             ▼
         分配任务      suri 问用户：
         /执行维护     "是否创建新角色？"
                          │
                     ┌────┴────┐
                     │         │
                    用户确认   用户拒绝
                     │         │
                     ▼         ▼
                suri 调用     返回用户
                role_manager  "无法处理"
                创建新角色
                     │
                     ▼
                新角色就绪后
                自己根据需求
                新增技能、调用
                MCP 工具完成
```

## 2. 单角色任务执行流

**场景**：suri 将任务分配给某个角色后，该角色的执行流程

```
suri 分配任务给角色
    │
    ▼
角色接收任务通知
    │
    ▼
角色分析需求
    │
    ├── 简单任务 ──▶ 直接执行（调用插件、MCP 工具等）
    │
    └── 复杂任务 ──▶ 调用 task_planner 分解任务
    │       │
    │       ▼
    │   生成 TaskPlan（步骤序列 + 依赖）
    │       │
    │       ▼
    │   调用 agent_registry.create_agent()
    │       │
    │       ▼
    │   按步骤执行循环
    │       │
    │       ├── 需要 LLM ──▶ 调用 llm_gateway
    │       ├── 需要工具 ──▶ 调用 mcp_framework
    │       ├── 需要记忆 ──▶ 调用 memory_service
    │       └── 纯逻辑 ──▶ 角色自行处理
    │       │
    │       ▼
    │   每一步完成 → agent_registry 更新状态
    │       │
    │       └── 受阻 → 触发 interrupt_handler
    │
    ▼
任务全部完成
    │
    ▼
角色汇总结果 → 返回给 suri / 直接返回给用户
    │
    ▼
触发 role_learner 异步学习（分析经验、检测技能模式）
```

## 3. 角色创建流程

**场景**：suri 发现没有合适角色时，向用户提案创建新角色

```
suri 分析需求 → 遍历角色能力索引 → 无匹配角色
    │
    ▼
识别能力缺口
    │
    ├─ 当前角色能力清单
    ├─ 缺口描述
    └─ 建议方案（创建新角色 / 扩展现有角色 / 使用通用工具）
    │
    ▼
向用户呈现提案：
"当前没有角色能处理'写产品文档'需求。
建议创建一个'文档撰写员'角色，我来帮她生成 Soul 草案。"
    │
    ▼
用户确认
    │
    ▼
suri 调用 llm_gateway 生成 Soul 草案：
  ├─ 昵称：根据需求生成
  ├─ 职责描述：完整职责范围
  ├─ 能力清单：capabilities
  ├─ 关键词：keywords
  └─ 工作方法论
    │
    ▼
向用户呈现完整 Soul 草案确认
    │
    ├── 确认 ──▶ suri 调用 role_manager.create_role()
    │               ├─ 创建目录结构
    │               ├─ 写入 soul.md
    │               ├─ 初始化 memories/
    │               └─ 注册到能力索引
    │
    ├── 修改 ──▶ 根据用户反馈调整草案
    │
    └── 取消 ──▶ 流程终止
    │
    ▼
发布 role.created 事件
    │
    ▼
新角色就绪，可被调度
    │
    ▼
新角色根据需求自己新增技能、调用 MCP 工具、完成任务
```

## 4. 角色自学与自增技能流

**场景**：角色完成任务后，通过 role_learner 分析经验并提升自己

```
角色完成任务 → 发布 task.completed 事件
    │
    ▼
role_learner 异步触发（不阻塞主流程）
    │
    ▼
读取角色记忆（最近 7 天 experiences）
    │
    ▼
LLM 分析（llm_gateway）
  ├─ 生成结构化洞察
  │     ├─ success_pattern — 成功模式
  │     ├─ improvement — 改进方向
  │     ├─ pitfall — 常见陷阱
  │     └─ preference — 偏好
  │
  └─ 检测技能形成模式（≥3 次重复的工具组合）
    │
    ▼
保存洞察到角色记忆
    │
    ▼
如有新技能模式 → 发布 role.skill_suggested 事件
    │
    ▼
suri 汇总并向用户呈现技能建议
    │
    ├── 用户确认 ──▶ 技能激活，写入角色 skills/
    ├── 用户修改 ──▶ 调整技能参数后激活
    └── 用户拒绝 ──▶ 标记为 ignored
    │
    ▼
下次任务时，相关洞察自动注入角色上下文
```

**角色也可以主动申请升级**：
```
角色发现现有工具不满足需求
    │
    ▼
生成新技能方案
    │
    ▼
通过 upgrade_manager 向 suri 提交升级请求
    │
    ▼
suri 评估 → 向用户呈现方案
    │
    ├── 确认 ──▶ 执行技能升级
    └── 拒绝 ──▶ 记录原因
```

## 5. suri 升级自身流

**场景**：suri 遇到自己也不知道怎么处理的需求时

```
suri 接收需求 → 遍历自己的技能 → 无法处理
    │
    ▼
suri 判断：既不是调度任务，也不是系统维护
    │
    ▼
suri 问用户：
"这个需求超出了我当前的能力范围。
是否允许我增加新技能来处理？"
    │
    ├── 用户确认
    │       │
    │       ▼
    │   suri 生成新技能方案
    │   ├─ 技能名称
    │   ├─ 触发条件
    │   ├─ 处理流程
    │   └─ 所需的工具/权限
    │       │
    │       ▼
    │   suri 向用户呈现方案
    │       │
    │       ├── 确认 ──▶ 技能激活
    │       └── 修改 ──▶ 调整后激活
    │       │
    │       ▼
    │   suri 更新自身技能列表
    │       写入 tool_descriptions.yaml
    │
    └── 用户拒绝
            │
            ▼
        记录原因，告知用户无法处理
```

## 6. 异常处理流

**场景**：角色执行任务失败或受阻

```
角色执行步骤失败
    │
    ▼
agent_registry.block_agent(reason)
    │
    ▼
发布 agent.blocked 事件
    │
    ▼
interrupt_handler 介入
    │
    ▼
分类失败原因
    │
    ├── missing_tool ──▶ 汇报 suri，请求开发
    ├── knowledge_gap ──▶ 搜索 / 升级给用户决策
    ├── permission_denied ──▶ 申请权限
    ├── dependency_failed ──▶ 自动重试（最多3次）
    ├── timeout ──▶ 自动重试 / 调整超时配置
    └── resource_exhausted ──▶ 等待 / 汇报 suri
    │
    ▼
生成用户决策建议
    │
    ▼
向用户呈现选项：继续 / 升级 / 取消
    │
    ├── 继续 ──▶ 重新调度
    ├── 升级 ──▶ suri 创建新 Agent 或增加新能力
    └── 取消 ──▶ agent_registry.cancel_agent()
```

**重试策略**：
- 退避间隔：[0s, 30s, 120s]
- 最大重试：3 次
- 仅 dependency_failed / timeout 类型可自动重试

## 7. 多角色协作流

**场景**：需要多个角色协作完成的复杂项目

```
suri 分析需求 → 识别需要多个角色
    │
    ▼
suri 创建项目组
  ├─ 指定项目总监（project_director 角色）
  └─ 分配各参与角色
    │
    ▼
项目总监调用 task_planner 生成协作规划
    │
    ▼
项目总监调用 task_scheduler 调度执行
  ├─ 并行步骤：同时调度多个角色
  │   └─ 各角色通过 role_comm 通信同步
  └─ 串行步骤：等待依赖完成后调度
    │
    ▼
项目总监定期播报进度
    │
    └─ 如有受阻 → 项目总监先协调，无法解决再汇报 suri
    │
    ▼
全部完成 → 项目总监汇总交付
```

## 8. 角色删除流程

```
删除发起（admin / suri / 用户）
    │
    ▼
检查目标角色类型
    │
    ├── core（suri）──▶ 拒绝（核心角色不可删除）
    │
    └── 其他类型
        │
        ▼
    检查活跃 Agent
        │
        ├── 有活跃 Agent ──▶ 等待完成 / 强制取消
        │
        └── 无活跃 Agent
            │
            ▼
        归档角色数据
          ├─ Soul → _archived/{role_id}/soul.md
          ├─ memories → _archived/{role_id}/memories/
          └─ skills → _archived/{role_id}/skills/
            │
            ▼
        发布 role.destroyed 事件
            │
            ▼
        清理运行时数据（30 天后自动物理删除）
```

## 9. 自优化上报流

**场景**：系统通过角色学习发现改进机会

```
role_learner / role 分析
    │
    ▼
生成 UpgradeReport / 技能改进方案
    │
    ▼
upgrade_manager.save_report()
    │
    ▼
状态: PENDING → SUBMITTED
    │
    ▼
suri 汇总，向用户呈现升级方案
  ├─ 变更原因
  ├─ 具体变更内容
  ├─ 影响范围
  ├─ 回滚策略
  └─ 风险评估
    │
    ▼
用户确认
    │
    ├── 拒绝 ──▶ upgrade_manager.update_status(REJECTED)
    ├── 延期 ──▶ upgrade_manager.update_status(DEFERRED)
    │
    └── 确认
        │
        ▼
    执行变更
        │
        ├─ 技能变更 → 热更新立即生效
        ├─ 代码变更 → 生成变更文件，IDE 模式应用
        └─ 插件变更 → 动态加载
        │
        ▼
    验证（健康检查 / 基础功能测试）
        │
        ├── 通过 ──▶ upgrade_manager.update_status(IMPLEMENTED)
        │
        └── 失败 ──▶ 执行回滚策略
