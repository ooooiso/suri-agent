# Suri Agent 迭代开发计划（v2）

> 基于全部 33 份 PRD 文档制定，核心目标：**让 suri 具备代码能力，能分析项目、制定编码计划、自我进化**。
>
> 代码能力作为 suri 的核心 skill，贯穿所有迭代逐步增强：
> - 迭代 1：读代码 → 分析项目 → 生成建议
> - 迭代 2：写代码 → 执行任务 → 运行测试
> - 迭代 3：自我分析 → 生成升级方案 → 执行代码变更
> - 迭代 4：从经验学习 → 提炼技能 → 持续进化
> - 迭代 5：扩展工具 → 完善运维 → 生产可用

---

## 迭代原则

1. **代码能力优先**：suri 的代码读写能力早于多角色协作实现
2. **PRD 驱动**：所有开发严格参考 PRD，实现与文档不一致时优先回归 PRD
3. **插件化**：所有功能以插件形式实现，无硬编码耦合
4. **零依赖**：仅使用 Python 标准库 + asyncio + SQLite
5. **安全沙箱**：所有代码操作受 security_service 审批和路径限制
6. **测试闭环**：每个迭代结束时必须有完整测试覆盖本迭代所有链路

| 迭代 | 目标 | suri 代码能力 | 新增插件 | 预计工期 |

## 代码能力与插件的对应关系

| 代码能力 | 实现插件 | 安全限制 |
|----------|----------|----------|
| 读取文件 | `code_tool`（迭代 1） | 只读，限制在 `suri-agent/`、`roles/`、`plugins/` |
| 列出目录 | `code_tool`（迭代 1） | 同上 |
| 写入文件 | `code_tool`（迭代 2） | 需审批令牌，禁止写入 `agent_framework/` 和 `suri_core` 除非用户确认 |
| 执行测试 | `code_tool` + test_framework（迭代 2） | 在临时目录运行，禁止操作生产数据 |
| 执行命令 | `code_tool`（迭代 2，简化） | 白名单命令（python、pytest、git status），禁止 rm/sudo |
| 自我分析 | `role_learner` / ProgramLearner（迭代 3） | 只读分析，不直接修改 |
| 生成升级方案 | `upgrade_manager`（迭代 3） | 用户确认后执行 |
| 应用代码变更 | `upgrade_manager` + `code_tool`（迭代 3） | 先备份、再变更、验证失败自动回滚 |

## 风险与回退策略

| 风险 | 应对 |
|------|------|
| suri 写错代码导致系统崩溃 | 所有写入需审批令牌 + 自动备份 + 验证失败回滚 |
| LLM 生成代码质量不稳定 | 先生成伪代码 → 用户确认 → 再生成完整代码 → 测试验证 |
| 文件沙箱被绕过 | AST 扫描 + 路径白名单 + 运行时权限检查 |
| 自我进化范围失控 | upgrade_manager 状态机强制用户确认每一步 |
| 迭代 1 范围膨胀 | code_tool 迭代 1 只实现只读，写入推迟到迭代 2 |
