# 插件与角色学习流程

> 本文档描述 suri-agent 中所有学习/自进化机制的完整流程。
>
> **核心原则**：学习是"分析→建议→确认→执行"的闭环。无实体可私自修改代码，所有变更须经用户确认。

---

## 一、角色学习流程（RoleLearner）

**定位**：单个角色的经验提取与技能进化。只分析一个角色的记忆，不跨角色，不分析框架。

### 触发条件

| 触发方式 | 来源 | 说明 |
|----------|------|------|
| 事件触发 | `task.completed` / `task.failed` | 角色任务完成后异步触发，不阻塞主流程 |
| 手动触发 | `user.command`（/learn role {role_id}） | 用户或角色主动发起 |
| 阈值触发 | 记忆积累达到阈值 | 预留机制，自动触发学习 |

### 执行流程

```
task.completed 事件
        │
        ▼
┌─────────────────┐
│ 1. 读取记忆      │
│    experiences  │
│    表（最近7天） │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 2. LLM 分析      │
│    生成结构化洞察│
│    · success_pattern
│    · improvement
│    · pitfall
│    · preference
└────────┬────────┘
         │
         ▼
┌─────────────────┐     ┌─────────────────┐
│ 3. 保存洞察      │     │ 4. 技能模式检测  │
│    roles/{id}/   │     │    重复工具组合   │
│    memories/    │     │    ≥3次 → 生成   │
│    insights/    │     │    技能建议       │
└─────────────────┘     └────────┬────────┘
                                 │
                                 ▼
                        ┌─────────────────┐
                        │ 5. 上报建议      │
                        │    role.skill_  │
                        │    suggested    │
                        │    → role_manager
                        └─────────────────┘
```

### 输出产物

| 产物 | 位置 | 内容 |
|------|------|------|
| 洞察文件 | `roles/{role_id}/memories/insights/{timestamp}_{type}.md` | 结构化分析结论 |
| 技能建议 | `role.skill_suggested` 事件 | 工具组合模式、使用场景、参数模板 |
| 上下文注入 | 角色系统提示（运行时） | 最近高置信度洞察，≤2000字符 |

### 上下文注入机制

```
角色接收新任务
        │
        ▼
role_learner.get_recent_insights_for_context(
    role_id={当前角色},
    task_hint={任务描述}
)
        │
        ▼
按时间(30天内) + confidence 排序
关键词粗排匹配 task_hint
        │
        ▼
选取 Top-N 洞察（总字符 ≤ 2000）
注入角色系统提示
```

## 二、插件学习流程（PluginSelfLearning）

**定位**：插件自我分析、生成升级建议。作为共享模块存在，各插件可选集成。

### 参与插件

理论上所有插件均可集成。重点关注：
- `llm_gateway` — 模型调用效率、失败模式、token 消耗优化
- `memory_service` — 查询性能、存储结构优化
- `mcp_framework` — 工具调用热点、权限策略优化
- `suri_core` — EventBus  worker 数、心跳策略、队列策略优化

### 执行流程

```
定时触发 / 事件触发 / 手动触发
        │
        ▼
┌─────────────────┐
│ 1. 数据采集      │
│    · 调用频次    │
│    · 错误日志    │
│    · 响应耗时    │
│    · 资源使用    │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 2. 模式分析      │
│    · 瓶颈识别    │
│    · 异常聚类    │
│    · 性能趋势    │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 3. 生成升级方案  │
│    · 变更原因    │
│    · 具体变更    │
│    · 回滚策略    │
│    · 风险评估    │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 4. 提交方案      │
│    plugin.upgrade│
│    _proposed 事件 │
│    → suri 角色   │
└─────────────────┘
```

### 进入统一升级流程

插件自学习生成的方案，进入**所有插件统一的自修改流程**（见第三章）。

## 三、全局分析流程（ProgramLearner）

**定位**：全局系统分析，由 **suri 角色**主动调用。不独立运行，不自动触发。

### 调用方式

```
suri 角色发布事件：
  event: task.created
  target: role_learner
  payload: {
    mode: "program_analysis",
    scope: "global",
    focus: ["plugin_efficiency", "role_collaboration"]
  }
```

### 执行流程

```
suri 角色调用
        │
        ▼
┌─────────────────┐
│ 1. 读取全局数据  │
│    · 事件日志    │
│    · 插件注册表  │
│    · 角色协作记录│
│    · 性能指标    │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 2. LLM 全局分析  │
│    · 插件调用热点│
│    · 协作瓶颈    │
│    · 资源使用模式│
│    · 架构优化建议│
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 3. 生成报告      │
│    返回 suri 角色│
│    event: task.  │
│    completed     │
└─────────────────┘
         │
         ▼
suri 角色决定：
  · 仅记录报告？
  · 生成某插件的升级方案？
  · 调整系统配置？
```

### 输出示例

| 分析维度 | 示例输出 |
|----------|----------|
| 插件调用热点 | "llm_gateway 占用了 73% 的事件流量，建议增加 worker" |
| 角色协作效率 | "suri → suri_dev 的任务转发平均耗时 2.3s，存在优化空间" |
| 资源使用 | "memory_service 在晚间出现峰值，建议增加缓存策略" |
| 架构建议 | "建议将 web_search 从 mcp_framework 内置服务拆分为独立插件" |

**注意**：ProgramLearner 只生成建议报告，不直接修改任何代码或配置。变更决策由 suri 角色做出，进入统一升级流程。

## 四、统一自修改流程（代码/配置变更）

**适用对象**：所有插件（包括 suri_core）、角色配置、系统配置。

**核心规则**：

1. 无实体可私自修改代码
2. 所有变更须经用户确认
3. 所有方案必须包含回滚策略
4. suri_core 升级后需通过冒烟测试

### 各实体变更权限

| 实体 | 可变更内容 | 变更方式 | 验证要求 |
|------|-----------|----------|----------|
| **普通插件** | 自身代码、配置 | 用户确认后热更新 | 健康检查 + 功能测试 |
| **suri_core** | 自身代码、配置 | 用户确认后，需额外验证 | 冒烟测试 + 健康检查 |
| **角色** | soul.md、技能、记忆 | 角色自进化或用户确认 | 无需代码级验证 |
| **系统配置** | config.json | 用户确认后热更新 | 配置校验 |

## 五、学习机制对比

| 维度 | RoleLearner | PluginSelfLearning | ProgramLearner |
|------|------------|-------------------|----------------|
| **分析对象** | 单个角色 | 单个插件 | 全局系统 |
| **触发方式** | 事件订阅（自动） | 定时/事件/手动 | suri 角色主动调用 |
| **输出** | 角色洞察 + 技能建议 | 插件升级方案 | 系统优化报告 |
| **是否修改代码** | 否（只生成数据） | 否（只生成方案） | 否（只生成报告） |
| **修改决策** | 技能建议上报，role_manager 决定 | 进入统一升级流程 | suri 角色决定 |
| **数据范围** | `roles/{role_id}/` | 插件自身目录 | 全局事件日志 + 注册表 |

## 六、事件流汇总

### RoleLearner

```
[角色完成任务]
    │
    ├── publish → task.completed
    │
    ├── subscribe ← role_learner
    │       │
    │       ▼
    │   分析角色记忆
    │       │
    │       ├── publish → role.skill_suggested → role_manager
    │       │
    │       └── 保存洞察到 roles/{id}/memories/insights/
    │
    └── 角色下次任务时 ← 注入相关洞察到上下文

### 技能建议 → 技能提升流程

```
role_manager 订阅 role.skill_suggested
    │
    ▼
评估技能建议
    │
    ├── 置信度 < 0.7 ──▶ 暂存，等待更多数据
    │
    └── 置信度 ≥ 0.7
        │
        ▼
    生成技能模板草案
        ├─ 技能名称
        ├─ 触发条件
        ├─ 工具组合
        ├─ 参数模板
        └─ 使用示例
        │
        ▼
    向用户呈现（access）
        │
        ├── 用户拒绝 ──▶ 标记为 ignored，记录原因
        │
        ├── 用户修改 ──▶ 更新草案，再次确认
        │
        └── 用户确认
            │
            ▼
        写入 skill 文件
            └─ roles/{role_id}/skills/{skill_name}.json
            │
            ▼
        更新角色 Soul 的 skills 列表
            │
            ▼
        发布 role.skill_invoked（首次激活）
```

**规则**：
- 技能提升需用户确认，不可自动执行
- 技能模板采用 JSON 格式，含 schema 校验
- 重复技能（名称相同）触发更新流程而非新建
- 用户可随时通过 `/skill disable {skill_name}` 禁用技能
```

### PluginSelfLearning → 统一升级

```
[插件自分析]
    │
    ├── 生成升级方案
    │       │
    │       ▼
    │   publish → plugin.upgrade_proposed
    │       │
    │       ├── subscribe ← suri 角色
    │       │       │
    │       │       ▼
    │       │   汇总，向用户呈现
    │       │       │
    │       │       ▼
    │       │   用户确认
    │       │       │
    │       │       ▼
    │       │   执行变更 → 验证 → 生效
    │       │
    │       └── subscribe ← suri_core（记录审计）
    │
    └── 变更完成 ← 发布 system.config_changed 或重启
```

### ProgramLearner

```
[suri 角色决策]
    │
    ├── publish → task.created (mode=program_analysis)
    │       │
    │       ▼
    │   subscribe ← role_learner / ProgramLearner
    │       │
    │       ▼
    │   全局分析
    │       │
    │       ▼
    │   publish → task.completed (报告)
    │       │
    │       ▼
    │   suri 角色接收报告
    │       │
    │       ├── 仅记录 → 结束
    │       │
    │       └── 生成升级方案 → 进入统一升级流程
    │
    └── 如进入升级流程 ← 同 PluginSelfLearning 流程
```
