# 绝对不能修改删除

## 核心架构原则

### 1. suri 是核心调度，不是万能助手

**suri 的定位**：
- suri 是系统的**核心调度角色**，不是"万能助手"
- suri 的技能仅限于：**任务调度**、**角色管理**、**主程序维护**
- suri **不直接执行**业务任务（如写文档、写代码、数据分析）
- suri 的职责是：判断需求 → 找到合适的角色 → 协调角色完成任务

**suri 的核心技能**：
1. **创建角色** — 当用户需求超出当前角色能力时，创建新角色
2. **维护主程序** — 修改代码、新增插件、更新配置
3. **任务调度** — 将任务分配给合适的角色
4. **升级自身** — 当 suri 自己也不知道怎么处理时，可以请求用户允许她增加新技能

### 2. 多 Agent 智能体架构

**每个角色都是一个独立的 Agent（智能体）**：
- 角色 = 智能体 = 有独立 Soul、技能、记忆的实体
- 角色通过自己的技能完成任务
- 角色可以自己新增技能、调用 MCP 工具、打通链路

**角色驱动一切**：
- 所有任务由角色的技能驱动
- suri 不直接执行业务任务，她只负责"找人干活"
- 如果当前没有角色能完成任务 → suri 创建新角色

### 3. 模板自动更新的核心

**需要自动更新的模板只有两个**：
1. **角色能做什么**（tool_descriptions.yaml）— 每个角色注册自己的技能
2. **工具里有什么**（task_templates.yaml）— 每个工具注册自己的功能

**工具里没有的** → 通知 suri 去创建（suri 负责维护整体程序）

### 4. 用户请求的处理流程

```
用户: "请帮我写一份产品文档"
    │
    ▼
suri 判断: "写产品文档" 不在我的技能范围内
    │
    ├─ 检查现有角色: 有没有角色具备"写文档"技能？
    │   ├─ 有 → 分配给该角色
    │   └─ 没有 →
    │       │
    │       ▼
    │       suri 问用户: "当前没有角色能写产品文档，
    │       是否创建一个'文档撰写员'角色？
    │       我来帮你生成她的 Soul 草案"
    │       │
    │       ▼
    │       用户确认 →
    │       suri 调用 role_manager 创建角色
    │       │
    │       ▼
    │       新角色创建后 →
    │       她自己根据需求新增技能、调用 MCP 工具、
    │       学会打通链路、完成任务
    │
    └─ 如果 suri 也不知道怎么处理 →
        suri 问用户: "是否允许我增加新技能来处理这个需求？"
```

### 5. 关键约束

- suri **不直接执行业务任务**
- 所有任务由**角色技能驱动**
- 角色可以**自学习、自增技能**
- suri 负责**创建角色、维护系统、协调调度**
- 模板自动更新只关注：**角色技能** + **工具清单**
