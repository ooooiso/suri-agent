# Agent 创建流程差距分析

> 分析日期: 2026-05-01
> 分析对象: `agent创建流程` 需求文档 vs 当前代码实现
> 分析范围: suri-agent/ 核心代码 + group/ 角色空间 + 基础设施

---

## 一、总体结论

**当前实现率约 25%**。基础框架（调度、任务状态机、权限、工具加载器）已搭建，但**所有涉及用户交互的创建流程、技能管理、工具开发闭环均未实现**。

| 模块 | 实现度 | 关键缺失 |
|------|--------|---------|
| 部门匹配与创建 | 10% | 无部门层级检查、无用户确认、无自然对话 |
| 角色匹配与创建 | 15% | 无角色缺失询问、无用户确认、ensure_core_roles 自动创建 |
| 技能赋予与匹配 | 5% | 无技能内容解析、无技能-需求匹配、无用户确认 |
| 工具调用 | 30% | 8 工具中只有 2 个有代码，参数校验为空实现 |
| 流程闭环 | 10% | Agent 计划生成但不执行，无重试，无用户同步 |

## 二、逐项对比分析

### 2.1 部门与角色的创建规则

#### 2.1.1 匹配检查

**文档要求：**
> Suri 收到需求后，首先检查是否存在可处理该需求的部门及部门负责人。
> 每个部门必须有一名明确的负责人。

**当前实现：**
- `cli.py::_detect_dispatch_target()` **直接关键词匹配角色**，跳过部门层级
- `task_dispatcher.py::_match_department()` 存在但被 `cli.py` 忽略，仅在 Telegram 模式使用
- `_match_department()` 的 keywords 是**硬编码 Python 字典**（design/engineering/ops/resource/hr/central），非从角色 Soul 动态推导

**差距：** ❌ 完全未按文档要求的"部门→角色"层级检查

#### 2.1.2 缺失处理 — 部门不存在

**文档要求：**
> 若没有匹配的部门：必须询问用户"是否创建新部门"，自然对话定义部门名称、职责、负责人。
> 必须得到用户明确确认后方可生效。

**当前实现：**
- `DepartmentRegistry.create_department()` 可创建部门，但**无用户确认流程**
- 无自然对话机制，无多轮交互
- `ensure_core_roles()` 在 `ConfigService.__init__()` 中**自动创建** 5 个核心角色，无用户干预

**差距：** ❌ 无询问、无自然对话、无用户确认

#### 2.1.3 缺失处理 — 角色不存在

**文档要求：**
> 若部门存在但缺少能处理需求的具体角色：必须询问用户"是否创建新角色"，自然对话定义名称、职责、所属部门。
> 必须得到用户明确确认后方可生效。

**当前实现：**
- 无角色缺失检测逻辑
- 无角色创建交互流程
- `suri_hr` 虽有 `role_creation` capability，但**无任何创建角色的代码逻辑**

**差距：** ❌ 完全未实现

#### 2.1.4 自然对话定义原则

**文档要求：**
> 创建过程不使用表单式提问，而是通过多轮对话逐步明确用户的期望，由 Suri 提炼并复述定义内容，待用户确认后写入系统。

**当前实现：**
- 无任何多轮对话创建逻辑
- 所有配置变更通过代码/文件操作完成

**差距：** ❌ 完全未实现

### 2.2 技能的赋予与工具调用规则

#### 2.2.1 角色存在但无匹配技能

**文档要求：**
> 当指派到一个"能做这类事"的角色，但其现有技能无法覆盖需求时：询问用户"是否为该角色增加此项技能"。用户确认后，该技能将记入角色的技能库。

**当前实现：**
- `ConfigService.list_role_skills(role_id)` 只扫描 `group/<role>/skills/` 下的**目录名**，不解析 skill.md 内容
- `cli.py::suri_process()` 角色执行时**不检查技能**，直接发送原始用户文本给模型
- 无"技能-需求"匹配逻辑
- 无用户询问流程

**差距：** ❌ 技能只存在于目录名层面，无内容匹配、无用户确认

#### 2.2.2 技能的自动实现途径

**文档要求：**
> 角色被赋予新技能后，优先自主调用"统一工具库"中的现有工具来兑现该技能。
> 如果统一工具库中存在可满足需求的工具，直接调用完成。
> 如果没有现成工具，提醒主程序/管理员：需要进行 MCP 工具调用开发，或独立工具开发。

**当前实现：**
- `ToolService` 有 8 个注册工具，但 **6 个只有 .md 文档，没有可执行代码**：
  | 工具 | 状态 | 说明 |
  |------|------|------|
  | file_read | ❌ 无代码 | 只有 file_read.md |
  | file_write | ❌ 无代码 | 只有 file_write.md |
  | file_list | ❌ 无代码 | 只有 file_list.md |
  | shell_exec | ❌ 无代码 | 只有 shell_exec.md |
  | db_query | ❌ 无代码 | 只有 db_query.md |
  | db_insert | ❌ 无代码 | 只有 db_insert.md |
  | model_manager | ✅ 有代码 | main.py 实现 |
  | web_fetch | ✅ 有代码 | main.py 实现 |
- `validate_params()` 是**空实现**（第 218 行 `TODO`），所有参数校验直接返回通过
- 无"技能→工具"的映射逻辑
- 无"无工具→提醒开发"的流程

**差距：** ⚠️ 工具框架存在，但 75% 的工具是空壳；无技能到工具的映射

#### 2.2.3 角色的自编程能力

**文档要求：**
> 角色自身可以调用编程工具能力，针对新技能编写建议性的执行脚本。
> 生成的脚本经审核后，存入该角色专属的技能库中，成为可复用的技能资产。
> 脚本的创建、修改必须遵循：版本记录、功能说明、触发条件等元数据规范。

**当前实现：**
- 无角色自编程逻辑
- `suri_hr` 有 `skills/templates/role_skills.md` 模板，但**无模板填充/生成逻辑**
- 无脚本审核流程
- 无版本记录机制

**差距：** ❌ 完全未实现（模板存在但无执行逻辑）

### 2.3 流程闭环

**文档要求的闭环：**
```
用户需求 → Suri 分析
    ├─ 有对应部门？
    ├─ 部门有对应角色？
    ├─ 角色有匹配技能？
    └─ 直接执行 / 自编程生成脚本 → 存入技能库
```

**当前实际流程：**
```
用户需求 → cli.py::suri_process()
    ├─ _detect_dispatch_target() 关键词匹配角色（跳过部门/技能检查）
    ├─ 创建 TaskPlan（ decorative，不执行步骤）
    ├─ suri 回复用户
    ├─ 逐个调度匹配的角色（直接发送原始文本给模型）
    └─ 保存经验 + 状态卡片
```

**差距：**
- ❌ 无部门层级检查
- ❌ 无角色缺失处理
- ❌ 无技能匹配检查
- ❌ Agent plan.steps 生成但不执行（装饰性）
- ❌ 无自编程流程
- ❌ 无技能入库流程

## 三、关键代码问题清单

### 🔴 P0 — 阻塞性问题

1. **6/8 工具只有文档没有代码**
   - 位置: `suri-agent/tools/*/scripts/`
   - 影响: 角色无法读写文件、无法执行 shell、无法操作数据库
   - 文件: `file_read.md`, `file_write.md`, `file_list.md`, `shell_exec.md`, `db_query.md`, `db_insert.md`

2. **Agent 计划生成但不执行**
   - 位置: `cli.py::suri_process()` 第 640–660 行
   - 影响: V3.0 的 TaskPlan 是装饰性的，`plan.steps` 存入数据库但从不逐个执行

3. **无用户确认机制**
   - 位置: 整个创建流程
   - 影响: 任何自动创建（角色、部门、技能）都违反文档要求

### 🟠 P1 — 严重问题

4. **部门匹配硬编码**
   - 位置: `task_dispatcher.py::_match_department()` 第 130–150 行
   - 影响: 新增部门需修改代码，无法动态扩展

5. **group_function.md 使用旧角色名**
   - 位置: `group/group_function.md`
   - 影响: 使用 `suri-hr`, `suri-dev`, `document-review`, `analyst` 而非 canonical 名称，可能导致调度匹配错误

6. **validate_params 空实现**
   - 位置: `tool_executor.py` 第 218 行
   - 影响: 工具调用无任何参数校验，类型错误/缺失必填参数会直接崩溃或产生不可预期行为

7. **TaskDispatcher 重试是空壳**
   - 位置: `task_dispatcher.py::handle_escalation()` 第 200–220 行
   - 影响: 失败任务只递增计数器，从不实际重试，也不通知用户

### 🟡 P2 — 中等问题

8. **_detect_dispatch_target 调用两次**
   - 位置: `cli.py` 第 620 行和第 748 行
   - 影响: 可能产生不一致的调度结果，浪费计算

9. **/tasks 命令未实现**
   - 位置: `cli.py::handle_command()` 第 246 行
   - 影响: 用户无法查看任务状态

10. **无配置热重载**
    - 位置: `ConfigService`
    - 影响: 角色/技能/工具变更需重启进程

11. **技能内容不解析**
    - 位置: `ConfigService.list_role_skills()`
    - 影响: 只返回目录名，不读取 skill.md 中的触发条件、执行步骤等元数据

## 四、核心流程规则（基于当前实现状态整理）

### 4.1 已固化的规则（已实现且可依赖）

**R1 — 空间分离规则**
- `group/`（角色空间）与 `suri-agent/`（主程序空间）严格分离
- 角色数据只存在于 `group/`，主程序代码只存在于 `suri-agent/`
- 违规由 `test_space_separation.py` 自动拦截

**R2 — 权限规则**
- 角色只能修改自己的 `group/<dept>/<role>/` 目录
- Soul 文件（`<role>/<role>.md`）由 `suri_hr`（admin）专属管理
- `suri_dev`（maintainer）管理 `suri-agent/` 代码
- `suri_hr`（admin）可管理所有 `group/` 路径

**R3 — 别名解析规则**
- 所有基础设施代码通过 `ConfigService.resolve_role_id()` 解析别名
- 别名不创建物理目录，数据始终写入 canonical 目录

**R4 — 任务状态机规则**
- `TaskService` 维护 `pending → in_progress → completed/failed` 状态流转
- 任务记录在 `suri` 的 role.db 中

**R5 — 输出路由规则**
- 角色输出通过 Soul `output_channels` 动态路由
- 输出路径由 Soul `output_path` 指定，归角色自有目录管理

### 4.2 需要补全的规则（文档要求但代码未实现）

**R6 — 创建确认规则（待实现）**
- 任何部门/角色/技能的创建必须经过用户明确确认
- 系统不可自行决定创建行为
- 自然对话是唯一的创建交互方式

**R7 — 部门优先匹配规则（待实现）**
- 调度流程必须先匹配部门，再匹配角色
- 部门匹配应从角色 Soul 的 `department` 字段动态推导，而非硬编码

**R8 — 技能匹配规则（待实现）**
- 角色执行前必须检查自身技能是否覆盖需求
- 技能不匹配时应询问用户是否增加技能，而非直接执行

**R9 — 工具完整性规则（待实现）**
- 统一工具库中的工具必须有可执行代码，不能只有文档
- 工具调用前必须进行参数校验
- 无现成工具时应进入开发排期并同步用户

**R10 — Agent 计划执行规则（待实现）**
- TaskPlan 生成的 steps 必须逐个执行，而非装饰性存储
- 每步执行结果应更新到任务状态
