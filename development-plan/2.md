> 版本: 3.0  
> 日期: 2026-05-01  
> 范围: 终端交互模式（CLI）下的多角色协同  
> 关联代码: `suri-agent/access/tui/cli.py`, `suri-agent/core/`, `suri-agent/infrastructure/config.py`

## 一、协同角色定义

| 角色 | 内部标识 | 类型 | 职责 |
|------|---------|------|------|
| **Suri（总调度）** | `suri` | scheduler | 用户唯一交互入口。需求分析、顶层任务分解、部门调度、最终结果反馈，协调创建部门/角色/技能/工具。 |
| **开发维护者** | `suri_dev` | maintainer | 主程序及工具的唯一维护者；代码演进、测试开发、基础设施维护。 |
| **人力资源** | `suri_hr` | admin | 部门划分、角色能力与协同管理；负责非核心角色的创建、挂载与生命周期控制。 |
| **审查专员** | `suri_review` | reviewer | 项目逻辑一致性把关；审查文件改动后的逻辑合理性，出具审查报告。 |
| **统计分析师** | `suri_stats` | specialist | 数据统计与项目指标持续优化；输出各类统计任务结果。 |
| **部门负责人** | 动态 | director/admin | 接收 Suri 分发的子任务，进行部门级需求分析与再分解，指派给具体执行角色。 |
| **执行角色** | 动态 | specialist/maintainer | 承担具体任务，自行分析并形成执行步骤，调用自身技能和工具库完成任务。 |

**核心约束**: 五个核心角色（`suri`, `suri_dev`, `suri_hr`, `suri_review`, `suri_stats`）不可删除。若缺失，系统自动重建默认 Soul 文件。

## 二、V3.0 任务状态机

### 2.1 Agent 生命周期

```
planning → running → [paused] → completed
    ↓          ↓
  blocked  (archived)
```

- **planning**: Agent 刚创建，步骤尚未开始执行
- **running**: 至少有一个步骤处于 `in_progress`
- **paused**: 用户手动暂停（预留）
- **completed**: 所有步骤完成
- **blocked**: 至少一个步骤受阻（依赖未满足、模型调用失败、角色未配置等）

### 2.2 TaskStep 状态流转

```
pending → in_progress → completed
   ↓           ↓
blocked    blocked
```

每个步骤包含:
- `step_id`: 唯一标识（如 `step_1`, `step_2`）
- `description`: 步骤描述
- `assignee`: 执行角色 role_id
- `depends_on`: 前置步骤 ID 列表（线性依赖或跨角色依赖）
- `result`: 步骤执行结果摘要（模型输出前500字符）
- `status`: pending / in_progress / completed / blocked
- `block_reason`: 受阻原因

### 2.3 计划生成策略

`TaskPlanService.generate_plan()` 根据匹配角色数量选择模板:

| 场景 | 策略 | 依赖关系 |
|------|------|---------|
| 单角色 | 使用该角色的能力模板（code/review/statistics/role_creation） | 线性依赖: step_i → step_{i+1} |
| 多角色 | suri 理解 → 各角色并行执行 → suri 汇总 | 角色步骤依赖 suri 理解步骤 |
| 无匹配 | 通用4步模板 | 线性依赖 |

**代码位置**: `suri-agent/core/task_plan.py`

## 三、调度链路（已实现）

### 3.1 完整调度流程

```
用户输入
  ↓
suri_process(text, user_id)
  ├── 1. 获取/创建会话 (session_id，24h复用)
  ├── 2. 创建任务 (task_id)
  ├── 3. 创建 Agent (agent_id)
  ├── 4. 检测调度目标 + 生成 TaskPlan
  ├── 5. 更新 Agent.steps
  ├── 6. suri 调用模型分析需求（注入动态调度规则）
  ├── 7. 输出 suri 回复 + 状态卡片
  ├── 8. 二次检测调度目标
  │      ├── 检查点1: 部门匹配
  │      ├── 检查点2: 技能匹配
  │      └── 检查点3: 计划步骤执行
  ├── 9. 结果回流（单角色/多角色汇总）
  ├── 10. 任务完成，保存经验卡片
  └── 11. 检测代码变更，自动刷新
```

### 3.2 双层调度匹配

`_detect_dispatch_target(text, suri_reply)` 实现:

1. **第一层（高优先级）**: suri 回复中明确包含角色 ID
   - 扫描 `suri_reply.lower()` 中是否包含 `suri_dev`, `suri_hr`, `suri_review`, `suri_stats`
   - 匹配到的角色直接加入调度列表

2. **第二层（关键词匹配）**: 用户输入匹配角色 Soul 中的 keywords
   - 遍历所有非 suri 角色
   - 读取角色 Soul frontmatter 中的 `keywords` 字段
   - 任一 keyword 出现在用户输入中即匹配

**代码位置**: `suri-agent/access/tui/cli.py` 第 1029-1062 行

### 3.3 动态调度规则注入

suri 的系统提示中动态注入所有角色的调度规则:

```
调度规则（你必须遵守）：
- 用户问关于你自身的问题 → 直接回答，不要调度
- 闲聊、问候、简单事实问答 → 直接回答，不要调度
- 用户问'程序维护、Bug修复、代码、重构...'等 → 由 suri_dev 处理
- 用户问'创建角色、组织架构、部门...'等 → 由 suri_hr 处理
- 用户问'审核、代码审查、审计...'等 → 由 suri_review 处理
- 用户问'统计、分析、报告...'等 → 由 suri_stats 处理
- 如果需求涉及多个角色，请明确列出所有相关角色名
```

规则从角色 Soul 文件 frontmatter 的 `keywords` 和 `capabilities` 字段实时读取，新增角色无需修改代码。

## 四、创建确认机制（CreationDialog）

### 4.1 三个检查点

在调度执行前，`suri_process()` 插入三个检查点:

| 检查点 | 触发条件 | 进入 CreationDialog |
|--------|---------|-------------------|
| 检查点1 | `_detect_dispatch_target` 返回空列表 + `_check_department_match` 返回 None | `start('create_dept')` |
| 检查点2 | `_detect_dispatch_target` 返回空列表 + 但匹配到部门 | `start('create_role', dept=...)` |
| 检查点3 | 角色无技能 或 技能 triggers 不匹配用户输入 | `start('add_skill', role=...)` |

### 4.2 CreationDialog 状态机

```
idle ──start()──→ creating(step=0: 确认是否创建)
                     ↓ 用户回答"是"
                  creating(step=1..N: 收集字段)
                     ↓ 所有字段收集完毕
                  confirming ──用户确认──→ [COMPLETE] → execute() → idle
                     ↓ 用户回答"修改"
                  creating(step=1: 重新收集)
                     ↓ 用户回答"取消"
                  [CANCELLED] → idle
```

**对话字段**:
- 创建部门: 名称 → 职责范围 → 负责人
- 创建角色: 名称 → 工作内容/职责
- 增加技能: 技能名称 → 功能和触发条件

### 4.3 执行创建

`_execute_creation()` 根据 action 执行:

| action | 操作 | 文件系统变更 |
|--------|------|-------------|
| `create_dept` | 创建 `group/<dept_id>/<dept_id>.md` | 部门定义文件 |
| `create_role` | 创建 `group/<dept>/<role_id>/<role_id>.md` + 子目录 | Soul 文件 + memories/skills/scripts/reference/output |
| `add_skill` | 创建 `group/<dept>/<role_id>/skills/<skill_id>/skill.md` | 技能定义文件 |

**关键约束**: 所有创建必须得到用户明确确认（"确认" / "是" / "yes"）后才写入文件系统。系统不可自行决定。

**代码位置**: `suri-agent/access/tui/cli.py` 第 1558-1615 行

## 五、计划步骤执行

### 5.1 `_execute_step()` 执行流程

```
1. 检查依赖: step.depends_on 中所有步骤状态是否为 "completed"
   └─ 否 → 标记 blocked，返回 False
2. 更新状态: pending → in_progress
3. 构建步骤专用 prompt（注入角色 Soul + 任务分解方法论）
4. 调用模型执行该步骤
5. 保存结果到 step.result（截断500字符）
6. 更新状态: in_progress → completed
7. 投递结果到终端
8. 广播状态更新到消息总线
9. 渲染并显示实时状态卡片
```

### 5.2 `_execute_dispatch()` 调度执行

```
1. 获取 Agent 和 steps
2. 过滤 assignee == matched 的步骤
3. 按 step_id 排序
4. 逐个调用 _execute_step()
5. 汇总所有步骤结果
6. 保存汇总结果到角色记忆
7. 返回汇总文本
```

**降级兼容**: 如果 Agent 不存在或 steps 为空，回退到一次性调度模式（V2.0 兼容逻辑）。

**代码位置**: `suri-agent/access/tui/cli.py` 第 1064-1238 行

## 六、能力缺口的处理闭环（已实现逻辑）

### 6.1 缺口检测时机

缺口检测发生在调度链路中，而非角色执行时:

1. **缺部门**: `_check_department_match()` 遍历所有部门，从角色 Soul 的 `keywords` + `capabilities` 动态推导部门关键词，无匹配则触发创建部门对话。

2. **缺角色**: 有部门匹配但 `_detect_dispatch_target()` 无角色匹配，触发创建角色对话。

3. **缺技能**: 角色匹配但 `list_role_skills()` 为空 或 `_skill_matches()` 返回 False，触发增加技能对话。

4. **缺工具**: 当前实现中，工具缺失不在调度前检测。角色执行步骤时若需要工具，由 `ToolExecutor` 负责加载。工具缺失时执行失败，步骤标记为 blocked。

### 6.2 缺口处理流程

```
用户输入
  ↓
调度目标检测失败
  ↓
定位缺口层级（部门→角色→技能）
  ↓
进入 CreationDialog 多轮对话
  ↓
收集名称/职责/描述等字段
  ↓
用户确认信息 → 写入文件系统
  ↓
刷新配置和部门注册表
  ↓
返回 "创建成功" 提示
  ↓
用户需重新输入原任务（当前未实现自动任务恢复）
```

> **当前限制**: 能力补齐后，系统提示创建成功，但尚未实现 "是否继续原任务？" 的自动恢复逻辑。用户需要手动重新提交原需求。

## 七、结果回流机制

### 7.1 单角色结果回流

`_summarize_result()`:
1. 构建汇总 prompt（suri 审阅角色结果 → 提炼关键结论）
2. 调用模型生成简洁中文汇报
3. 保存角色经验卡片
4. 返回汇总文本投递到终端

### 7.2 多角色结果汇总

`_summarize_multi_result()`:
1. 收集各角色结果（截断500字符）
2. 构建多角色汇总 prompt（按角色分段）
3. 调用模型生成整体汇报
4. 保存各角色经验卡片
5. 返回汇总文本

### 7.3 经验存储

每个角色完成任务后，自动保存经验:
- `role_id`: 执行角色
- `task_id`: 关联任务
- `action`: 任务摘要（前50字符）
- `result`: 结果摘要（前200字符）
- `feedback`: success / partial
- `tags`: task,dispatch / task,collaboration / task,complete

**存储位置**: 角色私有 SQLite DB (`group/<dept>/<role>/memories/role.db`) + 文件日志

**代码位置**: `suri-agent/infrastructure/memory.py`

## 八、辅助机制

### 8.1 代码变更检测与自动刷新

```
1. 初始化时计算 suri-agent/ 下所有 .py 文件的 mtime 哈希和
2. 每次任务完成后调用 _check_code_change()
3. 若哈希变化:
   - 首次: 提示用户 "输入 /reload 重新加载"
   - 任务完成后: 自动执行 _perform_reload()
4. _perform_reload() 使用 os.execv() 重启进程，保留角色记忆
```

### 8.2 会话隔离

多用户场景下通过 `user_id` 区分:
- 每个用户有独立的 `session_id`（24h 复用）
- 会话隔离消息历史、Agent 状态、记忆上下文
- `_get_or_create_session()` 管理会话生命周期

### 8.3 状态卡片

`StateCardRenderer` 实时渲染 Agent 状态:
- 单任务卡片: 显示任务名称、进度 (completed/total)、当前步骤
- 全局状态: 显示用户所有活跃 Agent 的摘要

## 九、关键约束（强制执行）

| # | 约束 | 实现方式 |
|---|------|---------|
| 1 | 用户交互唯一性 | 所有输出通过 `OutputRouter` 投递到 `TERMINAL` 渠道，role_id 标记为 `suri` |
| 2 | 创建必须用户确认 | CreationDialog 的 confirming 阶段必须收到 "确认/是/yes" 才执行 `_execute_creation()` |
| 3 | 核心角色不可删 | `SecurityService` 保护，仅 `suri_hr`（admin 类型）可修改核心角色 Soul 文件 |
| 4 | 别名自动解析 | `ConfigService.resolve_role_id()` 统一处理新旧别名（如 `suri-dev` → `suri_dev`） |
| 5 | 空间分离 | `group/`（角色空间）与 `suri-agent/`（主程序空间）严格分离，禁止同名嵌套 |
| 6 | 文档同步 | `DocSyncRule` 扫描代码变更后检查文档更新，`/sync` 命令生成同步计划 |
| 7 | 测试同步 | 新功能必须同时创建/更新测试，回归测试必须在修复前失败、修复后通过 |

### 10.2 数据模型

```python
@dataclass
class TaskStep:
    step_id: str
    description: str
    status: str = "pending"          # pending | in_progress | completed | blocked
    assignee: str = ""               # role_id
    depends_on: List[str] = []       # 前置步骤 ID

    agent_id: str
    task_id: str

    user_id: str = ""
```
