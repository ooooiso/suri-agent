

# Agent 项目全规则：开发态与运行态的彻底分离

本文档清晰定义以 **Agent 角色** 为核心的智能体项目在 **开发态（IDE）** 与 **运行态（Agent 运行时）** 的完整规则。开发态负责构建框架与工具，运行态负责在框架内自主协同、执行业务、自我进化。两种状态完全分离，不可互相通信，但通过预定义的接口单向衔接。

---

## 一、系统架构与核心理念

- **开发态**：人类开发者在 IDE 中编写全部代码，包括所有 Agent 角色（Suri、部门负责人、执行角色、HR、Suri_review 等）的逻辑、通用工具库、测试执行器等。开发者不能也不应该调用任何 Agent 角色来协助开发。
- **运行态**：编译部署后，系统启动，各类 Agent 实例化，通过 Suri 与用户对话，按照预置的协同规则动态创建部门、角色、技能，自主完成任务、更新文档与测试脚本。
- **核心衔接**：开发态为运行态**预埋**了所有必须的自助能力（权限控制、文档生成器、测试执行器、清单维护接口等），运行态 Agent 只需按规定调用这些能力，绝不接触源码或框架。

---

## 二、核心实体定义（共享理解）

| 实体 | 定义 | 维护者（运行态） |
|------|------|------------------|
| **部门** | 业务领域划分，必须指定一名部门负责人 Agent。 | HR 维护全局清单 |
| **角色** | 隶属部门，具备一组技能，在 `groups/<角色名>/` 下拥有独立空间。 | HR 创建骨骼，角色自维护内部文件 |
| **技能** | 角色可执行的能力单元，用自然语言定义，实现依赖工具或自生成脚本。 | HR 更新映射，角色维护实现脚本 |
| **工具** | 全局通用能力，由开发态提供基础实现，运行时可通过请求新增。 | 开发态实现并注册；运行态通过 Suri 请求开发 |
| **全局清单** | 记录所有部门、角色、技能、工具对应关系的文件，由 HR 在运行时维护。 | HR |
| **测试执行器** | 开发态提供的通用 CLI/服务，能让角色在沙盒内运行自己编写的测试并获取结果。 | 开发态实现 |

## 三、开发态规则（IDE 域）

### 3.1 工程文件与文件夹索引文档
- 每个文件夹下必须有一个与文件夹同名的 `.md` 文件，说明该文件夹总作用、与项目关系、子模块导航。
- **更新时机**：开发者每完成一个开发任务（新增模块、重构、修复），**必须立刻更新受影响文件夹的 `.md`**，确保工程文档与代码同步。
- 测试目录也同样遵循此规。

### 3.2 代码测试
- 任何源码改动，必须同步编写或更新**单元测试 / 集成测试**。
- IDE 应集成自动化钩子：提交前运行受影响测试套件，**全部通过方可提交**。
- 测试代码也是工程交付物，其质量与主代码同等要求。

### 3.3 为运行态预埋必须的能力（关键）
开发者须以代码实现以下支持机制，使 Agent 在运行态能自助运转：

1. **权限控制系统**  
   - 文件操作封装，确保每个运行态角色只能读写自己的 `groups/<角色名>/` 目录，以及通用工具库。  
   - `suri-agent/` 目录仅限 `suri_dev` 角色（或开发者）修改。  
   - 全局部门角色清单文件以及对其他角色目录文件夹的修正仅 HR 角色可写，用于修改角色定义。

2. **文档生成器**  
   - 提供工具或 API，在 HR 创建新角色时自动生成 `groups/<角色名>/` 下的标准目录和初始 `.md` 说明文件。

3. **测试执行器**  
   - 实现一个角色可在沙盒中调用的命令，能运行指定目录下的测试脚本，并返回标准化的通过/失败报告。  
   - 该执行器对角色隐藏底层实现，仅暴露调用接口。

4. **变更审核管道**  
   - 编程实现 `Suri_review` 的审核逻辑，包括：  
     - 检测变更文件的路径是否越权。  
     - 检查变更清单中是否包含测试结果及是否全部通过。  
     - 基于全局清单判断能力冲突风险。  
     - 输出标准化审核意见。

5. **全局清单维护接口**  
   - 为 HR 角色提供读写全局清单（部门、角色、技能、工具）的结构化 API 或文件操作限定。

6. **工具库注册与发现**  
   - 定义通用工具接口规范，使运行态技能可动态调用。  
   - 提供注册入口，以便新工具开发完成后能被运行态立即发现。

> 注意：上述能力均为**框架代码**，在开发态完成。运行态角色只是使用者，绝不可能修改这些框架。

## 四、运行态规则（Agent 协同域）

所有业务行为在此发生，Agent 严格遵循预置规则，绝无权限突破可能。

### 4.1 协同总流程
```
用户 → Suri（唯一入口）
         │ 需求分析，顶层分解
         │ 查全局清单，指派部门负责人
         ▼
   部门负责人（再分解 → 指派执行角色）
         ▼
      执行角色
         ├─ 有技能 → 调用工具/脚本执行
         └─ 无技能 → 上报 HR
                        │
                    HR 分析缺口（缺技能/角色/部门/工具）
                        │
                    Suri 对话用户确认创建
                        │
                    HR 更新全局清单
                        │
                    Suri 询问用户是否继续任务 → 重新调度
```

### 4.2 业务文档更新规则（说明文档）
- **何时触发**：创建新部门/角色/技能后；或角色新增/修改技能脚本后。
- **谁更新什么**：
  - **HR**：更新根目录全局清单文件（部门-角色-技能-工具映射）。
  - **各角色**：更新自己目录下的 `README.md` 或技能说明文档，详述新增或修改的技能、脚本用法。
- **审核与确认**：更新内容需整理成摘要，经 Suri 转 `Suri_review` 审核（主要检查与全局清单一致性），再由 Suri 呈现给用户确认后执行。

### 4.3 文件变更管理（角色私有文件）
- **何时触发**：角色在任务中通过编程工具生成新脚本、修改技能配置等任何对自己目录下文件的改动。
- **变更流程**：
  1. 角色准备**变更清单**（必含）：
     - 文件列表（路径、操作类型）。
     - 变更内容摘要。
     - 变更原因。
     - **测试结果附件**（见 4.4）。
  2. 将清单提交给 Suri → Suri 转交 `Suri_review`。
  3. `Suri_review` 自动审核：
     - 权限是否合法（所有变更文件均在角色目录下）。
     - 测试是否全部通过。
     - 是否可能破坏现有能力或产生冲突。
  4. Suri 将审核意见、变更摘要以自然语言告知用户，请求最终许可。
  5. 用户同意后，Suri 通知 `Suri_review`，再由其通知角色**正式提交文件变更**。
  6. 若涉及技能/工具新建或修改，角色须同时通知 HR 同步全局清单。

### 4.4 测试代码更新规则（角色自测试）
- **强制要求**：运行时角色开发，也存放在根目录 `tests/`框架 下。
- **执行**：调用框架提供的测试执行器，运行自身目录下所有测试。
- **通过标准**：只有所有测试全部通过，变更清单才可以提交审核。未通过或未包含测试结果的变更将被 `Suri_review` 直接否决。
- **异常处理**：若角色因沙盒环境限制无法运行测试（如缺少特定依赖），必须通过 HR 上报，并挂起变更，等待开发态解决环境问题。

### 4.5 工具缺口处理
- 当 HR 发现缺少必要工具时，形成“工具开发请求”提交 Suri，Suri 向用户转述：“需开发新工具 [描述]，工具就绪后方可继续”。用户同意后，该请求暂存。由于开发态与运行态分离，此时任务**挂起**，等待开发者完成开发、部署新工具后，用户或管理员通知 Suri 工具已就绪，HR 更新工具清单，任务才能被重新调度。

## 五、开发态与运行态的衔接界面

两者仅通过以下两个单向通道关联：

1. **部署通道**  
   IDE 开发完成并通过全部测试后，构建部署包，更新运行环境。新的 Agent 功能、工具、规则随之生效。运行态角色无法主动要求开发态“立刻修改代码”，只能通过“工具开发请求”沉淀需求，由开发者异步评估执行。

2. **工具请求队列**  
   运行态产生的工具开发请求以标准化数据格式（如 JSON 文件）存入指定位置，供开发者查阅。开发态完成工具开发并部署后，需在全局工具库中注册，使运行态 HR 可发现。

运行态**绝不能**触碰源码或工程文件；开发态**不能**直接调用正在运行的 Agent 服务来处理开发任务。二者在时间与空间上完全解耦。

## 六、完整场景举例

**用户需求**：“分析最新销售数据并生成 PPT 简报” （假设系统已有“数据分析部”，无“简报生成部门”）

- **开发态**：已提前开发好 Suri、部门负责人、HR、Suri_review 等角色的协同逻辑，实现了权限、测试执行器、工具库等。
- **运行态**：
  1. Suri 分解任务，发现需要“简报生成部”但不存在，遂询问用户创建。用户通过对话定义部门职责，并指定临时负责人。
  2. HR 创建部门及负责人角色，更新全局清单，文档生成器自动在 `groups/` 下生成对应文件夹和初始 `.md`。
  3. 数据分析部负责人正常分解分析子任务，由现有“数据分析师”角色完成。
  4. 简报生成部负责人指派“PPT生成师”角色（不存在），Suri 询问用户后创建。
  5. “PPT生成师”无对应技能，上报 HR。HR 发现工具库无 PPT 生成工具，提出工具开发请求。Suri 告知用户，任务挂起。
  6. 开发者接到工具请求，在 IDE 中开发 PPT 生成工具，编写测试，更新对应工程文档，部署上线，注册到工具库。
  7. 用户通知 Suri 工具已就绪（或系统异步通知），HR 更新工具清单，为“PPT生成师”添加技能。
  8. “PPT生成师”调用工具编写 PPT 生成脚本，同时编写测试脚本，运行框架测试执行器全部通过。
  9. 提交变更清单（含脚本和测试报告），经 Suri_review 审核、用户确认后，正式保存文件，HR 同步全局清单。
  10. 所有子任务完成，Suri 汇总回报用户。

整个过程，开发态和运行态全然分隔，仅通过一次工具开发请求产生异步连接。运行态所有操作均在预置规则内自主完成。

---

## 七、总结

- **IDE 开发**负责建造“世界”，提供 Agent 运行所需的全部规则与工具，自身不参与运行时业务。
- **Agent 运行时**在“世界”中依照既定法则自主协同，动态扩展能力，并通过审核和用户确认保证安全。
- 文档、测试、权限三条规则贯穿双态：开发态保证框架质量，运行态保证业务行为可追溯、可回滚、安全受控。

此规则集可直接转化为系统设计与开发任务清单，确保项目从构建到运行一路清晰、可控、可演进。
