# 多智能体协作框架（完整版）

## 一、框架概述

### 1.1 设计目标
构建一个**可扩展、自演化、人机协同**的多 Agent 系统，通过标准化的工具供给（MCP）与清晰的协作流程，实现复杂任务的自主分解、调度、执行与能力增长。

### 1.2 核心原则
- **唯一入口**：用户只与总调度 Agent 交互，其余 Agent 不直接暴露。
- **分层调度**：总调度 → 部门负责人 → 执行角色，层层分解。
- **能力自动修补**：执行过程中发现技能/角色/部门/工具缺口，由专门的 HR Agent 管理补齐闭环。
- **工具即服务（MCP）**：所有工具通过 MCP Server 标准化提供，Agent 动态发现与调用。
- **可并行**：任务分解、分配、执行均支持同步多 Agent。

## 三、智能体角色设计

### 3.1 总调度 Agent（Suri）
| 维度 | 说明 |
|------|------|
| **输入** | 用户自然语言指令 |
| **核心能力** | 意图理解、顶层任务分解、部门调度、结果验证与聚合 |
| **记忆** | 全局任务进度、部门能力目录、历史任务经验 |
| **必要工具** | 规划与分解引擎、部门目录查询 |
| **交互约束** | 是唯一可与用户对话的 Agent |

### 3.2 部门负责人 Agent
| 维度 | 说明 |
|------|------|
| **所属** | 一个逻辑部门（如研发部、数据部、合规部等） |
| **核心能力** | 将部门级任务二次分解为角色级任务，指派给执行角色 |
| **记忆** | 本部门角色清单、部门内任务队列、角色负载 |
| **必要工具** | 部门内角色查询、任务队列管理 |

### 3.3 执行角色 Agent
| 维度 | 说明 |
|------|------|
| **所属** | 特定部门的某一职能角色 |
| **核心能力** | 自主分析任务并形成执行步骤，调用技能/工具完成任务 |
| **记忆** | 自身技能库、执行历史、自我改进脚本 |
| **必要工具** | 全部 MCP 工具（按需调用）、脚本执行沙箱 |
| **技能库** | 可动态增加：任务脚本、Prompt 模板、工作流 |

### 3.4 HR Agent（能力管理）
| 维度 | 说明 |
|------|------|
| **职责** | 接收执行角色上报的能力缺口，分析缺口层级 |
| **核心能力** | 缺口定位（缺技能 / 缺角色 / 缺部门 / 缺工具），生成建议 |
| **记忆** | 全局角色清单、部门清单、技能清单、工具注册表 |
| **必要工具** | HR 管理数据库（创建、读取、更新能力条目）、通知 Suri 接口 |
| **约束** | 不直接面对用户，只与 Suri 和其他 Agent 通信 |

## 四、协同流程（完整闭环）

### 4.1 正常任务流
1. **用户请求** → Suri 接收。
2. Suri **顶层分解**，识别涉及的部门。
3. Suri 将子任务**并行派发**给对应部门负责人。
4. 部门负责人**二次分解**，指派给一个或多个执行角色。
5. 执行角色**自主规划步骤**，检查自身技能库：
   - **有技能** → 调用工具（MCP）执行 → 返回结果。
   - **无技能** → 跳转至 4.2 能力缺口处理。
6. 结果逐级上报，Suri 汇总后**回复用户**。

### 4.2 能力缺口处理闭环
1. 执行角色**上报 HR**，携带任务上下文。
2. HR **分析缺口**：
   - 缺技能：指定角色需新增技能
   - 缺角色：部门内无对应职能
   - 缺部门：整个部门不存在
   - 缺工具：技能所需工具未在 MCP Server 中
3. HR **反馈 Suri**，附带创建建议。
4. Suri **对话确认**：
   - 询问用户是否创建相应部门/角色/技能/工具。
   - 遵循创建顺序：部门 → 角色 → 技能 → 工具。
5. 用户确认后，**调用创建接口**（HR 执行），并更新全局清单。
6. HR 通知 Suri 补齐完成，Suri 问用户：“能力已就绪，是否继续任务？”
7. 用户确认后，任务**重新调度**，此时缺口已不存在。

## 五、工具体系与 MCP 集成

### 5.1 工具的分层
所有工具通过 **MCP Server** 统一提供，Agent 作为 MCP Client 动态发现和调用。

| MCP Server | 暴露的工具 | 用途 |
|------------|-----------|------|
| **Search Server** | `web_search`, `web_fetch` | 信息检索 |
| **File Server** | `read_file`, `write_file`, `parse_pdf` | 文件操作与解析 |
| **Database Server** | `sql_query`, `vector_search` | 结构化/向量数据访问 |
| **Code Interpreter Server** | `python_repl`, `shell_exec` | 代码执行与自动化 |
| **Communication Server** | `send_email`, `send_slack` | 人机协同通讯 |
| **HITL Server** | `request_approval`, `human_input` | 关键步骤人工审批 |
| **System Server** | `get_secret`, `schedule_task` | 密钥管理、定时任务 |
| **Media Server** | `generate_image`, `transcribe_audio` | 多模态处理 |
| **HR Internal API** | `create_role`, `register_skill` … | 能力管理操作 |

### 5.2 工具缺失处理
当执行角色发现完成任务需要某个外部功能，但 MCP Server 集群中没有对应的工具时：
- 执行角色上报 HR（缺口类型=缺工具）
- HR 反馈 Suri
- Suri 通知系统管理员或触发**自动 MCP 开发流程**（生成新的 MCP Server 配置/代码）
- 完成后注册到全局工具清单，任务恢复。

## 八、安全与治理

- **权限控制**：每个 Agent 分配角色，MCP Server 端按角色限制可调用工具。
- **人工审批**：关键操作（如发邮件、删除文件）必须通过 HITL Server 请求用户确认。
- **审计日志**：所有 Agent 的思考链、工具调用、任务流转全量记录。
- **内容安全**：输入输出经过审核工具（`moderate_text`, `moderate_image`），防止生成违规内容。

---

## 九、可观测性与监控

- **全局任务看板**：各 Agent 实时状态（待命/执行中/等待人审/报错）
- **追踪链**：单次用户请求的完整 Agent 调用链、耗时、Token 消耗
- **能力缺口告警**：HR 频繁触发缺口时通知管理员，优化部门/角色预设
- **MCP 健康检查**：监控各 MCP Server 的可用性、响应延迟

---

## 十、框架特点总结

- **分层负责，单向用户交互**：清晰的角色分工，用户体验简洁统一。
- **自我演化**：通过 HR 闭环，系统能够在使用中自动补齐能力短板。
- **工具标准化**：基于 MCP 的工具即服务，支持热插拔、独立演进、跨 Agent 共享。
- **安全可控**：全域审计 + 关键操作人审，满足企业级合规。
- **并行吞吐**：从顶层分解到执行均可并行，充分利用多个 Agent 的并发能力。

# 多角色智能体系统优化参考（V3.0 任务协同与多 Agent 架构）

## 1. 文档信息
- **文档名称**：多角色智能体系统任务协同与多 Agent 架构优化参考
- **版本**：V3.0
- **日期**：2026-05-01
- **关联系统**：suri 中枢智能体系统
- **前置依赖**：V2.0 多角色体系、昵称、自动重建、进化机制已明确

## 2. 优化目标
本次优化聚焦于**用户交互模式**、**任务执行透明度**、**并发任务管理**以及**跨部门协同**四大方向，旨在让系统更贴合“董事长与职能部门”的协作隐喻，同时支持复杂任务的高效分解与并行推进。

## 3. 用户交互层优化

### 3.1 唯一的对外出口：suri
- **规则**：在任何终端（命令行、Telegram 等）及对话界面，用户只能收到来自 `suri` 的消息。
- **其他角色**：不得直接向用户发送任何未经 `suri` 转述或封装的消息。
- **体感隐喻**：用户如同董事长，只能看到首席执行官（suri）呈报的信息，职能部门在其身后运转。

### 3.2 任务状态同步与步骤化播报
当用户向 `suri` 发起任务后，`suri` 每条回复必须包含**当前任务执行状态块**，格式如下：

```
【任务状态】
任务：<任务摘要>
总步骤：N步
当前步骤：第X步 - <步骤简述>
状态：<进行中|等待用户决策|已完成>
```

- **步骤分解**：`suri` 与部门经理（见下）规划任务后，将总步骤固定，并向用户一次性说明。
- **实时更新**：每当有进度变化（如某步骤完成、某角色遇到阻塞），`suri` 下次回复必须更新状态块。
- **终端 / Telegram 适配**：在群聊场景（中台播报），可由一个独立的“播报 agent”角色以标准化格式发布状态更新，但该角色也仅面向群聊输出，且其输出内容由 `suri` 授权。

### 3.3 中台播报（群聊环境）
- 若存在沟通群（如 Telegram 群），可由 `suri` 指令一个**播报角色**（如 `suri_broadcast`，非核心角色，由 hr 创建）定期同步各角色当前执行片段：
  ```
  【部门动态】
  - [小苏] 正在分析需求，整理架构调整建议...
  - [老李] 正在为主程序新增日志工具，进度 60%...
  - [数据小能手] 已生成初步统计报表，待审查...
  ```
- 播报角色仅发布与任务直接相关的进展，不透露内部敏感信息。

## 4. 任务规划与调度升级

### 4.1 部门经理角色（suri_dept_manager）
- **性质**：在五大核心角色之外，由 `suri_hr` 自动创建并维护的**中枢管理部门**，视作 `suri` 的规划副手。
- **内部标识**：`suri_dept_manager`
- **核心能力**：
  - 接收 `suri` 下达的任务意图，结合所有可用角色（核心及非核心）的能力矩阵，进行**任务步骤规划**。
  - 输出《任务执行计划书》，包含：步骤序列、每步负责角色/部门、预期产物、预估时间与依赖关系。
  - 计划书经 `suri` 确认后，由 `suri` 呈现给用户。
- **异常处理**：当某步骤执行失败、角色反馈无法完成或需要新能力时，部门经理需立即向 `suri` 提交中断报告及可行方案（如调用新角色、请求用户决策），由 `suri` 向用户汇报并获取授权。

### 4.2 中断与用户决策闭环
- 任何角色在执行中遇到以下情况，必须立即上报部门经理（或直接上报 `suri`，视实现路径）：
  - 超出自身能力范围
  - 需要用户输入、选择或额外资源
  - 出现预期外的技术阻塞
- `suri` 收到后，将用明确选项（如“A. 等待用户提供数据 B. 指派新角色尝试”）请求用户决策，更新任务状态为“等待用户决策”。

## 5. 多 Agent 并发与对话管理

### 5.1 新需求自动开启子对话
- 当用户在某个任务进行中提出**独立的新需求**（与当前任务上下文无明显依赖），`suri` 必须：
  - 将新需求生成一个独立的 **子对话 agent**（或称子任务上下文），与当前任务隔离。
  - 在回复中告知用户：“已为新需求开启独立追踪（任务ID: xxx），当前原任务继续执行。”
- 用户可随时切换查看不同任务的状态，`suri` 提供类似 `#任务ID 进度` 的指令入口。

### 5.2 任务内多 Agent 并行执行
- 部门经理在规划阶段即可将无相互依赖的步骤标记为**并行步骤**，并将它们分配给不同的角色（或部门）同时执行。
- 并行角色各自推进，执行完成后向部门经理汇报，部门经理汇总后更新总进度。
- 若并行任务间需共享中间产物，由部门经理指定唯一的中转角色（或共享存储契约）负责同步。

### 5.3 角色任务拆解方法论
每个角色（尤其是部门经理需要这一机制）在接到一项步级任务时，应内置一套**内部任务拆解方法论**，示例：
1. **目标复述**：用自己的话重新定义目标与成功标准。
2. **资源盘点**：梳理自身能力、可用工具、历史相似任务经验。
3. **子步骤生成**：将任务分解为可执行的原子动作（如“读取文件A → 分析结构 → 输出报告”）。
4. **依赖检查**：识别需要其他角色提供的前置产物。
5. **风险预注册**：预估可能阻塞点并提前准备应对策略。
6. **执行与自检**：每完成一个原子动作，自检输出是否符合预期。
- 该方法论应记录在角色的能力描述字段中，并作为进化对象（V2.0 进化机制的一部分），持续优化拆分粒度与准确度。

## 6. 中枢部门与扩展部门协同

### 6.1 中枢部门定义（五大核心）
- `suri`（调度）
- `suri_hr`（人力资源）
- `suri_dev`（开发）
- `suri_review`（文档审查）
- `suri_stats`（统计）
- 上述角色职责与进化机制按 V2.0 执行，新增的部门经理角色不属于核心不可删除名单，但为强建议保留角色。

### 6.2 HR 扩展部门机制
- `suri_hr` 负责根据任务需求动态创建**其他部门角色**（如前端开发部、数据分析部、测试部等），并在文件中注册。
- 每个新建部门需有：
  - 部门内部标识（`suri_dept_xxx`）
  - 能力描述
  - 下属角色列表（可由部门自行再细分）
- `suri` 发布任务时，可将任务指派给某部门负责人（即部门经理），由该负责人进一步调度其部门内角色。
- **核心流程**：`suri` → 部门经理 → 部门内角色执行 → 结果逐级上报 → `suri` 统一呈现。

## 7. 与原需求的兼容说明
- 昵称系统、自动重建、角色分离、进化机制等 V1.0/V2.0 内容全盘保持，本次优化仅补充任务协同与交互层面的定义。
- 部门经理的创建、消亡由 `suri_hr` 遵循非核心角色管理规则；其昵称与能力描述同样可配置。
- 所有交互原则不得突破“只有 `suri` 能与用户对话”的硬边界。

## 8. 下一步建议
- 开发部门（`suri_dev`）需要据此实现多会话管理模块、任务状态机、中台播报接口。
- 统计角色（`suri_stats`）需新增“任务并发效率”“步骤阻塞频率”等统计维度，反哺部门经理拆解方法论的优化。
