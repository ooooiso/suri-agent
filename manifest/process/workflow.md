# 工作流与自优化流程

## 1. 标准任务流

```
用户 → suri → 需求解析 → 匹配部门 → 下发总监
                                    ↓
用户 ← suri ← 汇总交付 ← 总监审核 ← 部门群指派 → 成员执行
```

### 详细步骤

1. **接收**：suri 接收用户需求，生成 `task_id`。
2. **解析**：suri 读取 `function_index.md`，匹配责任部门和总监。
3. **下发**：suri 向总监发送结构化任务消息（含 `task_id`、`需求`、`截止时间`）。
4. **拆解**：总监在部门群内拆解任务，指派给具体成员。
5. **执行**：成员按自身技能执行任务，过程中可向总监请求协助。
6. **审核**：成员完成后向总监汇报，总监审核质量。
7. **交付**：总监向 suri 交付最终结果。
8. **汇总**：suri 汇总（如涉及多部门则整合各方输出），向用户呈现。

## 2. 跨部门协作流

```
需求方总监 → 私聊提供方总监（抄送 suri）
    ↓
双方各自在部门内分派
    ↓
每 30 分钟或关键里程碑进度同步至 suri
    ↓
suri 整合后交付用户
```

### 强制规则

- 必须由**需求方总监**向**提供方总监**发起协作请求。
- 私聊内容必须抄送中台调度群 `@shushi_central_group`。
- 任何一方进度延迟，suri 自动介入协调。

## 3. 异常处理流

```
成员失败 → 重试（最多3次）
    ↓
重试耗尽 → 汇报总监
    ↓
总监无法解决 → 汇报 suri
    ↓
suri 向用户汇报 → 用户决策（重新规划/取消/升级）
```

## 4. 用户决策回路（问题回流）

开发人员实现中遇到无法自主解决的问题时，触发此回路：

```
开发人员遇到问题
    ↓
向总监汇报（本部门内尝试解决）
    ↓
总监无法解决 → 向 suri 汇报
    ↓
suri 整理问题上下文 + 可选方案 → 回流给用户
    ↓
用户判断"该功能如何实现"
    ↓
用户给出决策指令（方案 A / 方案 B / 重新规划 / 取消）
    ↓
suri 将决策下发给开发人员
    ↓
开发人员按用户决策继续实现
```

### 关键规则

- **回流必须带上下文**：开发人员汇报问题时，必须附上：
  - 已尝试的方案及失败原因
  - 卡住的具体技术点
  - 自己评估的 2-3 个可选方向
- **suri 不替用户决策**：suri 只负责整理和呈现，最终决策权在用户。
- **决策指令需明确**：用户回复需包含"采用方案 X"或"按以下方式实现：..."

## 5. 能力缺口处理流程

当用户需求无法匹配现有任何部门时，触发组织扩展评估：

```
用户需求输入
    ↓
suri 读取 function_index.md，遍历所有部门 function 字段
    ↓
匹配失败（无任何部门 function 与需求关键词重叠）
    ↓
suri 向用户汇报能力缺口：
  "当前平台暂无处理此需求的能力。"
  "现有部门：[列出所有部门及职能摘要]"
    ↓
suri 询问用户：
  "是否需要新建部门/角色来解决这个问题？"
    ↓
用户回复"是"
    ↓
suri 收集扩展需求：
  - 新建部门名称
  - 部门职能描述
  - 所需角色及技能
    ↓
suri 向 hr_admin 发起组织扩展请求
    ↓
hr_admin 按 role_management.md 创建流程执行
    ↓
新部门/角色上线 → suri 更新 function_index.md
    ↓
suri 重新调度原需求至新部门
```

### 关键规则

- **缺口必须明确**：suri 必须向用户清晰展示当前所有部门的职能边界，让用户判断是"需求描述不清"还是"确实需要新能力"。
- **用户驱动**：只有用户明确确认"需要新建"时才触发创建流程，suri 不主动提议扩展。
- **创建即启用**：新角色创建完成后，立即具备接收任务的能力，suri 将原需求重新调度。

## 6. 技能沉淀流程

用户确认的最终实现方案，沉淀为开发人员的新技能：

```
任务完成
    ↓
开发人员撰写技能总结（遇到了什么问题、如何解决、关键步骤）
    ↓
提交给 suri
    ↓
suri 提交 workflow_admin 审核
    ↓
审核通过 → suri 向用户确认
    ↓
用户回复"是"
    ↓
hr_admin 协助将技能写入 profiles/<role>/skills/<skill_name>/skill
    ↓
更新 profiles/<role>/skills/skills.md 索引
    ↓
git_admin 记录到 changelog.md
```

### 技能文件格式

```yaml
skill_id: "skill_xxx"
name: "技能名称"
origin: "从哪个任务/问题沉淀而来"
version: "0.1.0"
---

# 技能名称

## 触发条件
- （什么情况下会用到这个技能）

## 解决思路
- （用户决策的核心方案）

## 执行步骤
1. （步骤 1）
2. （步骤 2）

## 经验教训
- （遇到的问题和避免方法）
```

## 6. 自优化流程（原有）

角色完成工作后，可主动提出流程或技能优化：

### 优化报告格式

```yaml
optimization_id: "opt_xxx"
requester: "role_id"
timestamp: "..."
reason: "为什么需要优化"
before: "优化前的流程/技能描述"
after: "优化后的流程/技能描述"
impact:
  self: "对自身的影响"
  others: "对其他角色的影响（如有）"
  files: "涉及修改的文件清单"
```

### 审批链

1. **仅影响自身**：直接提交 `workflow_admin` → 分析通过后由 suri 向用户申请批准。
2. **涉及多角色**：需提前与相关角色同步并达成一致 → 再提交 `workflow_admin`。
3. **涉及文件修改**：还需满足 `security.md` 的变更审批要求。

### 约束

- `workflow_admin` 有权拒绝不合理优化，需说明理由。
- 用户拥有最终否决权。
- 优化通过后，由 `git_admin` 记录到 `changelog.md`。
