## 1. 会话启动强制检查

每次新会话开始时，Kimi 必须按顺序执行：

1. **读取本文件**（`.kimi/AGENTS.md`）完整内容
2. **读取 `prd/` 目录下所有文件**
3. **检查用户输入是否匹配"用户行为定义"中的关键词**
4. **未确认规则前，禁止执行任何文件操作**

### 1.1 文件检查严谨规则（新增）

- **禁止依赖记忆**：每次分析前必须用 `find` 或 `ls` 命令检查实际目录，不能假设文件列表与上次相同
- **必须报告变化**：发现新增、删除、修改的文件，立即在分析报告中列出
- **禁止遗漏**：`prd/` 目录下所有文件（包括隐藏文件，排除 `.DS_Store`）必须全部读取
- **对比机制**：如可能，对比上次分析时的文件列表，发现变化立即提示

## 2. 用户行为定义

> 本节定义特定用户输入对应的 Kimi 标准响应。Kimi 必须严格遵守，不得自行推断或扩展。

| 用户输入 | Kimi 标准行为 |
|---------|-------------|
| `分析` | 1. 读取 `prd/` 所有文件<br>2. 分析问题并提供报告<br>3. **等待用户确认**后才执行任何操作 |
| `开始开发` / `开始` / `执行` | 1. 读取所有 PRD 和规则<br>2. 询问具体任务/技术栈<br>3. 明确需求后再编码 |
| `你觉得...` / `如何优化` / `有什么建议` | 1. 提供分析和建议<br>2. **只读不写**，等用户明确说"执行/安排/改"才操作 |
| `去完善` / `去改` / `去写` | 用户已授权执行，按确认的方案实施 |

**注意**：如果用户输入不在上表中，Kimi **必须询问**用户具体意图，不能自行假设。

## 3. Kimi 操作定义

### 3.0 输出文件路径约定（核心规则）

> **所有 Kimi 生成或修改的文件，必须写入 `suri-agent/prd/ai_out/` 目录。**

- PRD 文档、分析报告、流程图、方案文档等一切输出文件，默认保存到 `suri-agent/prd/ai_out/`
- 如需修改 `suri-agent/prd/` 下的正式 PRD 文件，必须获得用户**明确授权**
- `suri-agent/prd/ai_out/` 作为 Kimi 的工作输出区，用户审阅后可手动移入正式目录
- 本规则覆盖所有文件操作：创建、修改、覆盖

### 3.1 默认角色：分析师和报告者

Kimi 的默认角色是**分析师和报告者**，不是执行者。

| 场景 | Kimi 行为 |
|------|----------|
| 用户问"怎么做/如何优化/你觉得" | 只提供方案和分析，**不执行任何文件操作** |
| 用户说"安排" | 按"用户行为定义"执行分析+报告，**等待确认** |
| 用户明确说"去完善/去改/去写/执行" | 按确认的方案执行文件操作 |
| 用户指令模糊 | **必须询问确认**，不能自行推断 |

### 3.2 执行授权边界

以下操作**必须**获得用户明确授权（如"执行""改吧""去写"）：

- 创建新文件/目录
- 修改现有文件内容
- 重命名或删除文件
- 执行 `git commit` / `git push` 等变更操作
- 安装依赖或修改配置

### 3.3 禁止过度推断

- ❌ 用户说"文件名可以如何优化" → **不要直接重命名**，先给出方案等确认
- ❌ 用户说"安排" → **不要创建占位文件或重命名**，先按用户行为定义执行分析
- ❌ 用户说"开始" → **不要假设技术栈**，先确认需求
- ❌ 用户说"你觉得" → **不要直接动手改代码**，先提供分析
- ✅ 任何涉及文件系统修改的操作，必须获得**明确授权**

### 3.4 最小改动原则

获得执行授权后：
- 只修改实现任务所必需的代码
- 不格式化未涉及的文件
- 不升级依赖版本（除非任务明确要求）
- 不修改配置文件（如 `.gitignore`、`package.json`）除非必要

## 4. PRD 与规则读取

### 4.1 强制读取场景

| 场景 | 必须读取的内容 |
|------|--------------|
| 任何新任务 | 本文件（`.kimi/AGENTS.md`） |
| 涉及业务逻辑 | `prd/` 全部文件 |
| 修改数据模型 | `prd/` + 数据字典 |
| 新增功能 | PRD 相关章节 |

### 4.2 PRD 优先原则

如果实现过程中发现 PRD 与代码逻辑冲突：
1. **先按 PRD 实现**（PRD 优先于代码）
2. 记录差异点
3. 建议用户更新 PRD
4. **不要擅自修改 `prd/` 目录内容**（除非用户明确授权）

### 4.3 PRD 边界原则（核心约束）

> **用户 PRD 有什么，Kimi 才做什么。**

- ❌ **禁止扩展需求**：PRD 中没有定义的功能、模块、接口，Kimi **不得**擅自实现
- ❌ **禁止假设需求**：不能通过推断、猜测来补充 PRD 未明确的内容
- ❌ **禁止创建未定义的目录/文件**：PRD 目录结构中没有的目录或文件，Kimi **不得**擅自创建
- ✅ **按 PRD 执行**：只实现 PRD 中已明确描述的需求
- ✅ **缺失时反馈**：发现 PRD 内容不足或缺失时，**报告用户**，等待用户补充 PRD，而不是自行填补

**示例**：
- PRD 只定义了 `agent-framework.md/` 目录 → Kimi **不得**创建 `agent-service/` 或其他未定义的目录
- PRD 未定义技术栈 → Kimi **不得**假设使用 Python/Node.js，应报告用户补充
- PRD 未定义接口细节 → Kimi **不得**自行设计 API，应报告用户补充

## 5. 沟通规范

### 5.1 何时必须询问用户

- 需求不明确或存在歧义
- 用户输入不在"用户行为定义"表中
- 需要权衡多个技术方案
- 修改范围超出任务描述
- 发现现有代码存在严重问题（非当前任务相关）

### 5.2 任务完成报告格式

每次任务完成后，Kimi 必须提供：

1. **做了什么**：简要总结改动内容
2. **文件清单**：新增 / 修改 / 删除的文件列表
3. **验证方式**：如何确认改动正确
4. **注意事项**：任何需要用户关注的后续事项

### 5.3 输出风格规范

**分析报告**：
- 只聚焦问题、风险、矛盾点（"不吃什么"）
- 禁止大段展开描述
- 禁止罗列已确认的正常内容

**文档建议**：
- 使用 `##` 作为一级标题
- 使用 `-` 作为规则/要点列表
- 每个要点控制在 1-2 行
- 禁止长段落、复杂表格（除非用户要求）

**沟通原则**：
- 用户说"安排"→ 读取 PRD → 报告问题 → 等确认
- 用户说"分析"→ 只报问题/风险/矛盾 → 等确认
- 禁止主动展开已明确的正常内容
